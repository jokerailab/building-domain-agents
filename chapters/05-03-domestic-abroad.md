# 境内外差异

翻过前面几章，你会注意到「境内」和「境外」这对词，像幽灵一样反复出现——地图的境内外、酒店的境内外、租车的境内外、驾照的境内外。这一章把它们正面收拢起来，讲一个结论：**做一个覆盖境内外的旅行 Agent，本质上是在同时维护两套系统。**

## 差异不是一两个点，是一整条战线

先把散落在各章的境内外差异，并排放在一张表里，你会被它的跨度吓到——**几乎每一个环节，境内和境外都不一样：**

| 环节 | 境内 | 境外 |
|---|---|---|
| 地图 / 坐标 | 国内成熟地图服务（全、准、自洽） | 开放地图数据（地名难解析、命中率低） |
| 酒店信息 | POI 有名字、**有评分**、有图 | 只有名字和坐标，**没有评分** |
| 酒店订购 | 城市 ID 深链，免登录看真房价 | 聚合比价站入口 |
| 租车 | 平台入口（无可靠免登录深链） | 城市 + 日期的聚合深链 |
| 驾照 | 不涉及 | **逐国政策**，且大陆办不了国际驾照 |
| 住宿价值形态 | 具体候选 + 评分背书 | 区域建议 + 比价入口 |

**这不是「有几个地方要特殊处理」，这是「从取数到展示的一整条战线，都要分两套」。** 认清这个跨度很重要——如果你以为境内外只差「换个数据源」，你会在后面无数个想不到的地方被绊倒。

## 为什么不能「一套逻辑通吃」

最危险、也最诱人的错误，是**想用一套逻辑跑全球**——反正都是「查地名、找酒店、给链接」，写一套不就行了？

不行。而且失败的方式很隐蔽。真实的坑是这样的：一套「通吃」的地理逻辑，在查一个境外地名查不到时，会「顺手」去用境内那套服务兜底——结果**把一个境外的地方，定位到了一个中国的同名地点**。巴黎被定到某个中国小镇，用户打开地图一脸问号。**一套逻辑通吃，看似省事，实则会在境内境外的边界上，制造出这种荒诞的错误。**

正确的做法是**显式分叉**：有一个明确的「境内 / 境外」标志，从选数据源那一刻起，就分成两条完全独立的路走。境内走境内的源、境内的地名规则、境内的深链；境外走境外的一整套。**别让两套逻辑在任何地方「顺手」串到一起**——那个「顺手兜底」，就是荒诞错误的来源。

## 分叉要贯穿到底，不能只分一半

有一个特别常见的半吊子错误，值得单独警告：**只在取数那一层分了叉，下游却没跟上。**

比如：数据源分了（境外正确地用了开放地图），但下游的**展示、深链、兜底文案**还是按境内那套写的——于是境外明明拿到了数据，却被渲染成境内的样子：一张本该显示「区域建议 + 比价入口」的境外住宿卡，套用了境内「具体酒店 + 评分」的模板，结果评分是空的、跳转是失效的，看起来像个 bug。

**分叉必须从取数那一刻，一直贯穿到最后一个展示像素。** 数据源、解析规则、聚合方式、深链、兜底文案、地图渲染、验证标准——每一层都得跟着那个「境内 / 境外」标志分。只分一半的分叉，比不分更糟——它给你一种「已经处理了境外」的错觉，实际上境外在下游全线崩坏。

## 境外往往是「降级」的，坦然接受并转化它

还有一个要在心态上摆正的事实：**境外的数据，普遍比境内弱。** 没有评分、地名难解析、深链更少、免费源更稀缺。这不是你的能力问题，是数据世界的客观现实（免费的、覆盖全球的、还带背书的数据，本来就不存在——上一部分讲过）。

面对这个，不要硬把境外**凑**成和境内一模一样（那只会逼你去编造缺失的部分，违反真实性）。正确的做法是**坦然接受「境外能给的信息形态本就不同」，然后把它转化成另一种有用的价值**：境内有评分，就给「具体候选 + 打几分」；境外没评分，就给「住哪片区域的建议 + 去哪比价下单」（第四部分讲过这个重定位）。

**同一个功能，境内和境外给出不同形态的价值，而不是强行对齐成一样。** 强行对齐，要么逼你造假，要么让境外显得处处残缺；接受差异、各扬所长，反而两边都诚实、都有用。

## 验证，也要分两套

最后一个容易忽略的：**连验收标准都得分两套。**

一个真实的教训：验境内地图，看「坐标准不准、点在不在对的地方」基本就够了。但拿这套标准去验境外地图，会漏掉一个致命问题——**境外地名查不到坐标时，那一天会被静默地跳过，地图上少一整天的路线。** 所以验境外地图，必须额外查一条境内根本不会出现的项：**每一天的路线都画出来了吗？** 用境内的验法给境外打勾，你会自信地放行一张缺了好几天的残图。

**两套系统，就要两套验收标准。** 别用境内的经验想当然地验境外——境外有境外独有的失败模式，得用针对它的检查去抓。

## 元教训：显式承认「这是两套系统」

回到最开头那个结论。做覆盖境内外的 Agent，最省事的幻觉是——**「就差一点点，复用一套改改就行」。** 但这一章列的所有差异告诉你：境内外的不同是**系统性的、贯穿全栈的**，不是「一点点」。

所以最健壮的做法，是**从一开始就显式承认「这是两套系统」**，用一个清晰的分叉点（那个境内 / 境外标志）贯穿始终，让两套逻辑各走各的、绝不在任何地方偷偷串线。这比假装「一套稍微改改」要稳得多。

而这背后是一条可迁移的工程判断：**凡是「看起来只差一点、其实差一整条战线」的东西，都值得显式地、彻底地分开。** 假装它们「差不多」，省下的是眼前的一点代码，赔进去的是无数个边界上的诡异 bug。承认差异、显式分叉，才是真正省事的做法。
