# 子 Agent 与并行

前面几章讲的都是**一个**模型、在**一条**主循环里干活。但真实的专用 Agent，几乎一定会走到这一步：有些活，交给主循环顺手做并不好。

- 要核实五个垭口的路况，一个个串着搜，慢；
- 搜出来一堆网页噪音，全塞进主上下文，脏；
- 让刚排完方案的模型自己检查方案，它会倾向于「我排得挺好」，不客观。

**子 Agent 就是为这三件事而生的：并行、上下文隔离、客观性。** 它对标的是 Claude Code 里的 Task / 子 agent——主 Agent 把一件明确的活派出去，子 Agent 独立干完，只把**结论**交回来。

野行 Y 里有两个真实的子 Agent，正好各解决其中几个问题。

## 子 Agent 一：research（并行 + 隔离）

排一条自驾环线，模型常常一次要核实一批事实：折多山 10 月封不封路、稻城亚丁最佳季节、雨崩徒步几天、某个网红观景台是不是真的存在……

早期这些是靠模型一个个调 `web_search` 串着查的。问题有两个：**慢**（五个点串行，等五轮）；**脏**（五次搜索的原始网页摘要全堆进主上下文，就是上一章「上下文膨胀」的元凶）。

`research` 工具（`app/agent/tools.py`）把这件事改成了一个并行子 Agent。模型一次把要核实的一批问题交给它：

```python
def _research(args):
    topics = args["topics"]          # [{key:"折多山", query:"折多山垭口 10月 封路"}, ...]
    # 1) 并行搜索：一批 query 同时打出去，不再一个个等
    with ThreadPoolExecutor(max_workers=6) as ex:
        got = list(ex.map(lambda t: web_search(t["query"]), topics))
    # 2) 蒸馏：一次 LLM 调用，把每个主题的搜索摘要压成一句事实结论
    verdicts = distill(got)          # 折多山 → "10月通常可通行，遇强降雪偶有临时管制"
    return {"results": verdicts}     # 只有结论回主循环，原始网页留在子 Agent 里
```

它干了两件事，正好对应子 Agent 的两个价值：

**并行**。五个点同时搜，墙上时间从「五轮之和」压到「最慢那一轮」。实测四个点并行约 9 秒，串行要四倍。对一个用户在旁边等着看行程一点点长出来的产品，这个差别是体感级的。

**上下文隔离**。这是更妙的一点。原始的五坨网页摘要，全部消化在子 Agent 内部——主循环的上下文里，只回来五句蒸馏过的结论。**噪音留在子 Agent，干货回主循环。** 这跟上一章「上下文工程」是一体两面：与其让主上下文膨胀了再回头压，不如从一开始就让脏活在一个隔离的地方做完，只把结论端回来。子 Agent 天然就是一道上下文防火墙。

这也是 Claude Code 派 Task 子 agent 去「搜整个代码库找某个用法」时的同一个道理——你不想让几十个文件的内容涌进主对话，你只想要那句「它在这三个地方被调用」。

## 子 Agent 二：critic（客观性）

第二个子 Agent 解决的是完全不同的问题：**客观性。**

模型排完一条主线，正要拿给用户确认。这时候有个风险——它可能漏了用户的硬约束：用户说「怕高反」，它却排了个 4700 米垭口还连着长途；用户带着老人，它却让某天开 8 小时；三个人结伴，其中一个人的必去点被悄悄漏掉了。

你不能指望**排方案的那个模型**自己揪出这些。它刚生成完方案，上下文里全是「我为什么这么排」的理由，让它自查，它倾向于自我确认——「嗯，我排得挺合理」。这是 Agent 的一个通病，也是人的通病。

Claude Code 的解法叫 verify-before-done：完事之前，用一次独立的核对确认真做对了。野行 Y 照这个思路加了 `_critique_plan`（`app/agent/loop.py`）——一个**换了脑子**的核对子 Agent：

```python
def _critique_plan(messages, summary, provider):
    intent = _user_intent(messages)              # 抽出用户在对话里表达的所有约束
    prompt = f"""【用户需求】{intent}
【Agent 排的主线】{summary}
逐条核对，只报确凿硬伤：
1. 硬约束是否违背：怕高反却上高海拔？带老人小孩却单日暴走？预算档不符？说要去的点漏了？
2. 多人出行：把每个人分别列出来，逐个核对他的必去与诉求有没有照顾到。
3. 里程驾时是否离谱：某天 >500km 或驾时 >6h 没拆分？环线是否真闭合？
4. 天数是否匹配：排不下（太赶）？或有空转的一天（太松）？
只报真问题。"""
    return llm(prompt)["issues"]                 # 有硬伤 → 打回重排；没有 → 静默放行
```

几个设计上的讲究：

**它是「无工具」的一次纯核对。** 不搜、不算，就对着「用户要什么」和「方案是什么」逐条比。角色单一，它就不会跑偏去干别的。

**它换了一套上下文。** 它拿到的是重新抽取的用户约束 + 主线摘要，而不是继承主 Agent 那一大堆「我的推理过程」。**新的脑子、干净的视角**——这正是它比主 Agent 自查更客观的原因。

**它接在 checkpoint（给用户确认）那一步，不是最后出报告那一步。** 为什么？因为这个产品的卖点之一是「主线秒出、报告先看」，如果把核对压到最后会拖慢出报告。放在确认卡之前，既拦住了硬伤，又不牺牲速度——**有问题的主线，用户根本看不到**，它已经被打回重排了。

**它只回弹一次。** 打回去让主 Agent 改一版、重新核对，就放行——`critic_bounces` 上限为 1。核对是为了兜住硬伤，不是追求完美到把用户晾在那儿反复改。这是「质量」和「延迟」之间一个刻意的取舍。

验证过它真管用：一个「怕高反却安排 4700 米 + 长途」的方案，它揪出 4 条真硬伤；而一个「成都轻松 3 天」的正常方案，它 0 误报、静默放行、不打扰。**方案本身好的时候，它一声不吭；有硬伤时，它在用户看到之前拦下来。**

## 什么时候该派子 Agent

两个例子背后，是一套可迁移的判断。当你在主循环里遇到下面任一情况，就该考虑派个子 Agent 出去：

| 信号 | 为什么主循环不该自己做 | 子 Agent 提供 |
|---|---|---|
| 一批同类的活（核实多个点） | 串行慢 | **并行** |
| 会产生大量中间噪音（搜索、扫库） | 污染主上下文 | **隔离**（只回结论） |
| 要客观评判自己刚产出的东西 | 自我确认偏差 | **换个脑子**核对 |

有一个成本要认：子 Agent 不是免费的——它多花一次（或几次）模型调用，也多一层延迟和复杂度。所以别滥用。判据很简单：**这件活如果留在主循环里做，是不是会让主循环变慢、变脏、或变得不客观？** 三个「是」里中一个，就值得派出去；一个都不中，就别为了架构好看而拆。

还要记住一个前提：**子 Agent 的质量，取决于它用的模型。** 这个项目里 critic 用的是主模型链，一旦主力模型降级到弱模型，critic 的核对质量会同步下降——它可能漏判、也可能误报。子 Agent 不是魔法，它只是把「同一个模型」用在了「更合适的角色和时机」上。模型本身的健康，是另一部分的主题。
