# 模型分层

第一部分讲的是 Agent 的骨架——循环、工具、上下文、记忆、交互。骨架搭好了，得给它一个「大脑」。但「用哪个模型」不是一个选择题，是**一组**选择题：排行程骨架用什么、并行核实的蒸馏用什么、写个分享文案用什么、主力挂了拿什么兜底——**不同的步骤，值得配不同的模型。** 这就是模型分层。

## 先把「一次模型调用」抽象出来

在讲分层之前，得先有一层抽象，让「换模型」这件事本身是廉价的。

这个项目里，整个模型层只暴露一个动作——跑一轮对话：

```python
class LLMProvider:
    def chat(self, messages, tools=None) -> ChatTurn: ...

@dataclass
class ChatTurn:
    text: str
    tool_calls: list[ToolCall]
    usage: dict
    model: str          # 这一轮实际是哪个模型服务的（给 UI 显示）
```

底下挂两个适配器：`OpenAICompatAdapter`（通义 / Kimi / GLM / OpenAI 这些走 OpenAI 兼容格式的）和 `ClaudeAdapter`（Anthropic SDK，另有 thinking / effort）。工具用**一套中性的 JSON schema 定义**（第二章讲过的四元组），每个适配器把它翻译成自家的线格式——因为 Anthropic 和 OpenAI 都吃 JSON Schema，这层翻译是机械的。

这层抽象的意义，和「工具即中性四元组」是同一个道理：**上面的循环完全不知道底下是哪家模型。** loop 只管调 `chat()`，拿回一个 `ChatTurn`。想换模型、换厂商、加一条 fallback 链、接一个网关——全在这层底下折腾，循环一行不改。没有这层，分层和 fallback 都无从谈起。

## 模型不是可互换的零件

接下来是这一章最重要、也是我用真实翻车换来的一个认知：**不同模型不是同一个东西的不同档位，它们的能力是有下限的，而且这个下限会咬人。**

具体说：排一趟行程要靠模型**连续调工具**（function-calling）——搜、估、核实、提交，一步接一步。强模型能稳稳地驱动这个循环；但掉到某些弱模型（比如免费兜底的 `glm-4-flash`），它**驱动不了**——不解析你已经给的参数、把问过的问题反复问、内容浅、甚至编地名、路线来回折返，最后流程卡死、报告出不来。

第一次遇到这种情况，我的本能反应是「代码回归了，去查 bug」。查了半天发现——**代码没问题，是那一轮恰好 fallback 掉到了弱模型。** 这些「变蠢」的现象，是模型的能力下限，不是工程缺陷。

这件事给我一条铁律，写进了项目记忆：

> 一旦链头掉到弱模型，智能类测试全被污染（不解析参数、ask 死循环、内容浅、编地名）——这些是**模型能力下限，不是回归，别当 bug 修**。先看 UI 上的模型徽章确认当前到底跑的是哪个模型，再下结论。

所以这个项目里，每一轮跑完，`ChatTurn.model` 会带回**真实服务这一轮的模型名**，显示在 UI 徽章上。为什么要费这个劲？因为**当你分不清「模型变蠢了」还是「代码坏了」，你会浪费大量时间修一个根本不存在的 bug。** 把「当前是哪个模型」变成一个随时看得见的事实，是分层系统的一个必备仪表盘。

## 分层：强、快、兜底

认清模型不可互换之后，分层的逻辑就自然了——**按任务的「难度」和「频率」，把它配给合适的层。**

这个项目大致分三层：

| 层 | 特点 | 干什么 |
|---|---|---|
| **强** | 贵、稍慢、能稳定 function-calling | 排行程骨架、critic 核对——要连续调工具、要判断力的重活 |
| **快** | 便宜、快、够用 | research 的蒸馏、分享文案生成——单轮、不需要复杂工具编排的轻活 |
| **兜底** | 弱、但永久免费在线 | 前两层全挂时接住流程，别断线 |

关键在于**把任务和层匹配对**。用强模型去跑「把五段搜索摘要压成五句话」（`research` 的蒸馏），是杀鸡用牛刀，慢又贵；用快模型去驱动整个排行程的工具循环，它扛不住，流程会崩。分层不是「越强越好」，是**让每个任务落在能干好它的最便宜的那一层**。

举个这个项目里的真实例子：`research` 子 Agent（上一部分讲过）并行搜完之后，那一步「蒸馏成每点一句结论」用的是普通的 `make_provider()`——一个不需要顶配模型的轻活；而主循环里排骨架、出报告，走的是能扛 function-calling 的强链。同一个系统里，不同步骤跑在不同层上。

## 兜底层的铁律：宁可弱，不可断

三层里，兜底层有一条独立的原则值得单说：**兜底模型的第一要求不是「聪明」，是「永远在」。**

链尾那个 `glm-4-flash` 又弱又糙，但它有一个别的模型没有的优点——**永久免费、不限额、几乎不会挂**。当强模型撞了限流、快模型因访问受限连不上，它是最后那道「至少别让用户对着一个崩溃页面」的防线。

这背后是一个取舍判断：**一个偶尔变糙但从不断线的系统，体验好过一个大部分时候很聪明、但关键时刻白屏的系统。** 兜底层的存在，就是把「最坏情况」从「彻底失败」抬到「勉强能用」。它不追求好，它追求「有」。当然，兜底不能是无声的降级——用户至少该被告知「现在引擎有点忙」，这属于弱模型护栏的范畴，后面单独讲。

## 对专用 Agent 意味着什么

如果你要做一个专业领域的 Agent，模型分层是你在「大脑」这一层的第一个设计决策。三条可迁移的经验：

1. **先建一层 provider 抽象**，让换模型是廉价的——否则后面的分层、fallback、网关全都无处安放；
2. **别把模型当可互换零件**，实测出你的每个模型的能力下限，把「当前跑的是哪个」变成看得见的仪表盘，省下修假 bug 的时间；
3. **按任务难度和频率分层**，让每个步骤落在能干好它的最便宜那层；兜底层只求「永远在」。

---

> 分好了层，还有三个问题没解决：这些模型怎么串成一条链、主力挂了怎么自动换下一个、以及——在境内部署时，怎么保证这条链**连得上**（有些海外模型源在境内访问受限）。下一章讲**模型网关与 fallback 链**，以及一个通用架构书几乎不会提、但做本土专用 Agent 绕不开的约束：可达性。
