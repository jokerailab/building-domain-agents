# 机票

行程要接上真实机票。但机票很快会教你一件事：**这是一个没有完美数据源的领域。** 这一章讲的，不是「怎么接一个完美的机票 API」——那个不存在——而是**在每个数据源都有硬伤的现实下，怎么做出一个诚实、可用的机票能力。**

> **先说清一个贯穿这一部分的前提：下面所有关于数据源的「拿不到、没有、凑不齐」，都是建立在「坚持用免费、可用的数据源」这个前提上的。** 如果愿意花钱，数据几乎没有买不到的——真实成交价、全球覆盖、带评分的酒店库，商业数据服务都有。但那是另一套成本账。这个项目的选择是：**先把免费、可达的数据源用到极致，把数据成本压在近乎零。** 所以后面说的「拿不到」，都是指「免费拿不到」，不是「世界上不存在」——这个边界，请一直记着。

## 没有一个完美的机票数据源

理想的机票数据源长这样：官方、全、免费、带航班号和时刻、境内能访问。**现实是，这样的源不存在。** 挨个试下来，每一个都在某个维度上不合格：

- 有的是**官方源，但内测邀请制**——没有自助申请入口，普通开发者根本拿不到 key；
- 有的**能申请，但免费额度小或对国内航线弱**；
- 有的**数据好，但境内访问不了**（又回到上一部分的可达性问题）；
- 有的**免费可用，但只返回价格，没有航班号和时刻**——那用户没法「选具体哪一班」。

**做专用 Agent，你会反复遇到这种「没有一个选项是完美的」的局面。** 这时候纠结「哪个最好」是没意义的，因为没有最好的。真正要做的，是**想清楚你的判据，然后在一堆有缺陷的选项里，挑那个缺陷你能接受、优点你最需要的。**

## 选源的判据：不是「最全」，是「够真、可达、能诚实」

对机票，我的判据不是「数据最全最准」，而是四条底线：

1. **免费 + 境内可达**——上一部分的硬约束，不满足直接出局；
2. **带航班号和时刻**——因为产品要让用户「比多班、选一班」，只有价格是不够的；
3. **国内外都覆盖**——这个产品境内境外行程都做；
4. **能诚实标注**——数据是「参考」性质，产品得能如实说明，而不是假装它是成交价。

按这四条筛下来，最后选定的源（Travelpayouts）不是「最权威」的，但它免费、境内可达、国内外全覆盖，而且——**它返回的不是一个价格，是多个真实航班**（航班号、时刻、航司、直飞还是中转、价格）。这一点是关键：**「比价」的前提，是先拿到多个真实的、可区分的航班。** 有了多航班，才能在 `choose` 里列出来让用户比、让他挑一班非最低价但时刻更好的（第一部分「持续交互」讲过这个闭环）。

**选数据源，本质是选「你最不能妥协的那个维度」。** 对机票，我最不能妥协的是「真实、可比、能诚实标注」，而不是「最权威」。想清楚这个优先级，选择就不纠结了。

## 「参考价」三个字的分量

机票有一个和别的商品不同的特性：**它的价格是实时浮动的**——同一班飞机，此刻和五分钟后、这个舱位和那个舱位，价格都可能不一样。这意味着——**没有任何一个数据源，能给出「你此刻点下去的确切成交价」。**

面对这个，诚实的做法是标「**参考价**」，而不是把一个查来的数字伪装成「你要付的钱」。这三个字看着小，分量不轻：它是在对用户说「这个价帮你心里有个数，但以你下单时的实际为准」。

这是上一章「真实性」在机票上的直接体现——**标注不确定性，是一种品德。** 用户其实完全能接受「参考价」：他要的本来就是「大概多少钱、心里有个数、知道去哪订」，而不是一个会在下单时被打脸的精确数字。**假装精确，比诚实地模糊，更伤信任**——因为前者会在成交那一刻当场穿帮。

## 一个只有真跑过境外行程才知道的坑

最后讲一个非常具体、非常「领域」的坑，它是那种在办公室里想不到、只有真让 Agent 跑一趟境外行程才会暴露的问题。

境外行程的机票，出发地（origin）**必须是一个真实的中国出发城市**。但模型有个坏习惯——它会给出发地填一个占位符，比如「国内出发地」。结果这个占位符没法解析成机场代码，机票查询直接返回空，报告页出现一张空白的机票卡。

解法是明确约束：**境外行程的 origin = 你从中国哪个城市飞出去，hub = 境外落地的第一座城市。** 「从成都飞曼谷」，origin 是成都、不是「国内」。

这个坑的价值不在坑本身，在它揭示的一个道理：**领域里最致命的往往不是大问题，是这种「看起来无关紧要、实则让整个功能失效」的小细节。** 「出发地别填占位符」——这句话你在任何机票 API 文档里都找不到，它只能从「真的让 Agent 排一趟境外行程、看着那张空白卡」里得到。**专用 Agent 的打磨，一大半就是在填这种只有真跑起来才会露出的小坑。**
