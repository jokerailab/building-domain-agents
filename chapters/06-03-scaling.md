# 并发能力

上一章算出了容量上限——一个不大的两位数并发。这一章讲，怎么在**不推翻架构**的前提下，把这个上限顶高、把系统抗突发的能力硬化。会看到，很多抗压能力不是靠「重写架构」换来的，而是靠一组**配置级**的加固——风险低、见效快，是小系统最划算的抗压手段。

## 先看瓶颈：单 worker 下最稀缺的资源

要硬化，先得知道卡在哪。单 worker 这个架构下，最稀缺的资源是**线程**——具体说，是被 SSE 长连接占住的线程。

一个排行程的 SSE 连接，从建立到结束要好几分钟（模型一步步跑、前端一点点收流）。这几分钟里，**它一直占着一个线程。** 普通的 HTTP 请求几十毫秒就还回线程了，而一个排行程会话，占着线程不放好几分钟——这让线程成了整个系统里最先被耗尽的东西。所以抗压硬化，主要就是围绕「怎么让有限的线程和连接扛住更多并发」来做。

## 连接池：别用默认值

数据库连接池，是第一个该显式配置、而不是用默认值的地方。

默认的连接池大小，往往不匹配你的实际场景。这个项目显式设了几件事：

- **池大小和溢出上限**——明确规定常驻多少连接、峰值最多能再开多少，而不是听凭默认；
- **连接回收**——设一个回收时限，让连接用一阵就换新的，避免长期空闲的连接悄悄失效（数据库那头可能早把它掐了，你这头还以为能用）；
- **获取超时**——拿不到连接时别无限等，等一小会儿就报错，把问题暴露出来，而不是让请求默默挂死。

还有一个容易忽略的边界：**如果数据库是多个应用共享的，你的池上限要留余量给别人。** 你把连接占满了，别的应用就连不上了。**连接池不是「越大越好」，是「够用且不挤占邻居」。**

## 线程池：异步框架的隐藏瓶颈

第二个该显式调的，是**线程池**——这是异步框架里一个特别隐蔽的瓶颈。

异步框架把同步的活（还有上一章说的、那些不能阻塞事件循环的重活）丢进一个线程池执行。而这个线程池的**默认大小可能很小**。前面算过：一个排行程占大约两个线程，如果线程池默认只有几十，那除以二，二十来个并发就把它占满了——**后面的请求只能排队等线程，整个站点开始发卡。**

解法是把线程池的容量显式调大（比如翻一倍），容量直接跟着翻倍。但这里有一个**很关键的坑**：这个线程池的大小，在有些框架里是一个「运行时变量」，**必须在事件循环已经启动之后、在异步的启动钩子里去设**——在普通的同步初始化代码里设，是不生效的。这个坑不知道的话，你会以为自己调大了线程池，实际上根本没生效，然后对着「怎么还是卡」百思不得其解。

## SSE 长连接：又贵又必须，得精打细算

SSE（服务端推送）是 Agent 流式输出的必需品——「边做边说」「一步步长出行程」全靠它。但它是个**长连接**，一个连接占一个线程好几分钟，比普通请求贵得多。既然它又贵又不能不用，就得精打细算地伺候：

- **排队也占线程，排队要封顶。** 第二部分的并发闸门讲过——排队中的连接同样占着线程，所以队列必须有上限，否则一次流量高峰，光是排队的人就能把线程池占满，反而拖垮全站。
- **空闲的连接别白占资源。** 前端做「省电守卫」：用户把标签页切走时，就暂停那些轮询、动画、心跳——别让一个用户看都不看的页面，在后台继续占用连接和计算。**对长连接的产品，「用户没在看的时候别干活」是一条省资源的基本纪律。**

## 硬化的边界：配置级 vs 架构级

把上面这些串起来看，它们有一个共同点——**全是「配置级」的加固：调池子的大小、封队列的上限、掐空闲的开销。** 它们不动架构、风险低、当天就能见效。

而更大的扩容手段——多 worker、给模型池配多个 key 做负载均衡、把会话状态挪到外部存储——是「**架构级**」的。它们能顶更高的量，但复杂度也高一个量级（多 worker 要解决会话粘性、外部会话要解决一致性）。

**这两者有明确的先后：先把配置级能榨的榨干，再谈架构级。** 因为配置级的加固，常常已经能把容量翻倍、扛住当前阶段的量，而代价只是改几个参数。**在配置级还没榨干之前就上架构级扩容，是拿一大笔复杂度，去换一个你其实用配置就能拿到的容量——不划算。**

## 别乱加「扩展性」

最后回到上一章「简单胜过正确」的那条线。这个项目明确守着一条：**在没真正到瓶颈之前，别盲目引入多 worker、分布式这些架构级的东西。**

因为单 worker（会话在内存）+ 配置级硬化 + 并发闸门，这套组合已经把一个小系统的抗压能力，做到了和它当前规模相称的程度。**过早地上架构级扩容，是拿实实在在的复杂度，去换一个你现在根本用不上的容量上限——而那份复杂度，从落地那天起就要你天天为它买单。**

**「能扛住现在、且知道下一步往哪扩」，比「一步到位扛住想象中的未来」健康得多。** 前者让你轻装前进，后者让你背着一套用不上的复杂架构，在还没有那么多用户的时候，先把自己拖慢。
