# Agentic Loop：模型驱动的循环

做专用 Agent，第一个要想清楚的架构决策是：**它是一个「循环」，不是一条「流水线」。**

这个区别听起来抽象，但它决定了整个系统的形态。绝大多数人第一次做「AI 规划行程」，第一反应是搭一条流水线：

```
用户输入 → 提取目的地 → 查机票 → 查酒店 → 排每天 → 拼成行程 → 输出
```

这条流水线能跑，但它很脆。因为真实的规划过程**不是线性的**：

- 用户说「想去高原自驾」，你得先**问**他几天、几个人、什么车——但也可能他一句话里就说全了，不用问；
- 查到稻城亚丁回成都要开 8 小时，你得**回头**把这天拆成两天、加一个住宿点；
- 查机票时发现某个目的地根本没有直飞，你得**换**一个思路，或者告诉用户；
- 用户中途改主意「不要高原了，换海岛」，整个流程得**重来**，但又不能全丢。

流水线处理不了这些「回头」「换路」「重来」。你会写满 if-else，最后变成一坨没人敢动的状态机。

## 循环是怎么工作的

Agentic loop 换了个思路：**不预先规定步骤，让模型自己决定下一步做什么。**

循环的骨架非常简单：

```
while 没完成:
    模型看当前上下文 → 决定调哪个工具（或说完成了）
    执行工具 → 把结果塞回上下文
    回到循环开头
```

模型每一轮只做一个决定：「基于现在知道的一切，下一步该干什么？」可能是「搜目的地」，可能是「问用户几天」，可能是「估算这段驾时」，也可能是「行程排好了，提交」。工具的结果回到上下文里，模型看到结果，再决定下一步。

这就是 Claude Code 的核心，也是野行 Y 的核心。在这个项目里，这个循环体现在 `app/agent/loop.py` 的驱动逻辑里——它不关心「第几步该做什么」，只是反复地：把上下文喂给模型、拿到工具调用、执行、把结果喂回去，直到模型调用 `submit_plan`（提交行程）。

## 为什么循环能处理「回头」和「换路」

因为循环里没有「步骤」的概念，只有「当前状态」和「下一步决定」。

- 用户中途改主意？新的消息进上下文，模型下一轮自然会基于新意图决定动作，不需要「重置流水线」。
- 驾时太长要拆天？模型看到 `estimate_leg` 返回的 8 小时，下一轮自己决定加一个住宿点——这不是你写死的规则，是模型的判断。
- 某个目的地没直飞？工具返回「无数据」，模型看到后换策略。

**规则从代码里搬到了模型的判断里。** 你不再维护一个庞大的状态机，你维护的是「给模型什么工具、什么上下文、什么约束」。这是心智负担的根本转移。

## 循环不是放任，需要护栏

但「让模型自己决定」不等于放任。一个裸的循环会出两类问题：

1. **不收敛**——模型一直调工具，绕圈子，不提交。得有**步数上限**和**强制收尾**：接近上限时提示模型「该出结果了」。
2. **跑偏**——模型漏掉了用户的硬约束（预算、天数、忌讳）。这个项目里加了一层 **checkpoint**：关键节点让一个「审查子 Agent」核对约束有没有被满足，没满足就打回。

所以真实的 loop 不是 `while True` 那么天真，它是：**模型驱动 + 护栏兜底**。护栏包括步数上限、强制收尾、约束核对、异常重试与降级（模型挂了换一个）。这些会在后面「模型层」和「持续交互」两章展开。

## 对专用 Agent 意味着什么

如果你要做一个专业领域的 Agent，第一件事不是去设计流程，而是想清楚三件事：

1. **这个领域的「动作」有哪些**——它们会变成你的工具（下一部分「工具设计」讲）；
2. **模型需要看到什么才能决定下一步**——这是上下文工程；
3. **什么约束不能违反**——这是护栏。

流程本身，交给循环。**你定义能力边界和护栏，模型负责在里面把活干完。** 这是专用 Agent 和传统「规则引擎 + 模板」最本质的区别，也是它能处理真实世界那些「回头、换路、重来」的原因。
