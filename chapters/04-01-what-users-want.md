# 用户到底要什么

前三部分讲的是「怎么把 Agent 造出来」——架构、模型、工具。从这一部分起，转向一个更根本、也更容易被工程师忽略的问题：**这个 Agent 到底在帮用户解决什么？**

做专用 Agent 最容易搞错的一步，恰恰是这个——**「以为自己知道用户要什么」。** 工程师容易把「用户要一份旅行规划」理解成「用户要一段关于旅行的文字」。但这两者差得很远。这一章讲，旅行用户真正要的是什么。

## 用户要的是「能直接出发」，不是「一段文字」

你让通用 Agent「规划一条川西自驾」，它能给你一段很漂亮的文字：稻城亚丁、色达、四姑娘山，配上「建议 7 天」。读起来很专业。但用户真要照着走，立刻卡壳——**这段文字没法直接变成行动。**

因为用户真正要的，不是「读一段介绍」，而是一份**能直接出发**的东西：

- 一张**地图**——到底去哪、怎么连成一条线；
- 每天**住哪**——不是「找个地方住」，是具体到落脚点；
- 每段**开多久**——今天这段路 5 小时还是 9 小时，一天到不到得了；
- 大概**花多少**——机票、住宿、车，心里有个数。

**「一段文字」和「一份能出发的行程」之间，隔着的正是所有专业工作。** 通用 Agent 停在文字，专用 Agent 要落到「能行动」。用户衡量你的产品，不是看文案写得美不美，是看「我能不能拿着它就走」。这是第一条，也是最根本的一条：**产出必须是可执行的，不是可阅读的。**

## 信息聚合器：关键信息要能「看到、能比、能选」

顺着「能出发」往下，还有一层：用户做决策时，需要的信息要**摆在眼前**，而不是「你帮我定了就行」。

这个项目把自己定位成一个**信息聚合器**——「有哪些酒店、什么航班、什么车」这些关键决策信息，要让用户在对话和报告里**直接看到、能比较、能选择**，而不是藏在一个「相信我，我帮你订好了」的黑箱里。**看不到，就没有竞争力。**

举个具体的：查机票，不是只告诉用户「大概两三千」，而是列出几班真实航班——航司、航班号、时刻、直飞还是中转——让他自己比、自己选哪班。只有到「下单」那一步，才跳去外部平台。**决策的信息在我这儿摆全，交易的动作交给平台。**

这背后是对「用户要什么」的一个判断：**用户要的不是被替他做决定，是被给足做决定的信息。** 一个把信息藏起来、只给结论的 Agent，用户不敢信（他没法验证）；一个把真实选项摊开、让他自己拍板的 Agent，用户才有掌控感。当然，这些摆出来的信息每一条都必须是真的——这就接回了上一章的真实性：**能看到 + 都是真的，这两条一起，才构成「可信的信息聚合」。**

## 自驾用户，有一套独特的需求

再往细里走。「旅行」是个大词，而这个 Agent 服务的是一个更具体的人群——**自驾的人**。自驾用户有一套通用旅游 Agent 根本不懂的需求：

- **他要环线**——从一个地方出发、绕一圈回来，不走回头路。环线是自驾的招牌玩法，规划一条漂亮的环线，是硬功夫。
- **他在意路况**——这条路烂不烂、垭口封没封、季节对不对，直接决定能不能走。
- **他有驾时的忍耐上限**——一天开 5 小时和开 11 小时是两回事。排出「一天开 12 小时」这种腿，就是不懂自驾。
- **他常想去小众、轻越野的地方**——大路货的景点，他自己会开车去，用不着你。

**这些需求，通用 Agent 给不了，因为它没把自己代入「一个真的要开车上路的人」。** 它不知道「稻城到成都单天开不完，得中途住一晚」，因为它没有「驾时」这个概念。专用 Agent 的价值，一大半就在于**把这个细分人群独有的、通用产品照顾不到的需求，一条条接住。**

## 最独特的需求：多个人的「共识」

但如果只讲到「服务好一个自驾用户」，还漏了这个领域最独特、也最难的一个需求——**自驾常常是几个人、几台车结伴出行。**

一群人一起自驾，每个人偏好不同：有人想徒步、有人怕高反、有人要住好点、有人预算紧。这时候用户要的**不是「服务提问的那一个人」，而是「撮合出一版大家都能接受的共识行程」。** 这是普通旅行 Agent 完全不做的一件事——它们默认只有一个用户、一个人的偏好。

所以这个 Agent 专门为「多人」设计了能力：

- 识别出这是多人出行（用户提到「我们几个」「两家人」或人数偏多），就主动考虑分歧；
- 平衡冲突——节奏快慢不一就折中或某天自选，必去点冲突就都排进不同天，体力差异大就给分组方案（「想徒步的去雨崩两天，其他人在镇上等」）；
- 在给用户确认主线时，摆一张「**谁 · 想要什么 · 这版怎么照顾到**」的小表，让每个人一眼看到自己的诉求有没有被排进去；
- 出行程时专门讲清**这一版是怎么平衡各人诉求的**（`consensus` 字段）。

核心判断是一句话：**共识 > 某一个人满意。** 别让一个人的偏好压过其他人。

**这是野行 Y 最独特的一块——它服务的不是「一个用户」，是「一群要一起出发的人之间的那份共识」。** 这个需求，通用 Agent 够不到（它没有「多人」的概念），穷游发帖也够不到（帖子是单向的、不撮合）。找到这样一个「别人结构上做不到、而你能做」的需求，是专用 Agent 立身的关键。

## 怎么知道用户要什么

最后，怎么才能知道这些？不是靠拍脑袋，也不是靠一份问卷。

一半靠**把自己彻底代入真实场景**——真的去想「一个人要带着一家老小自驾川西，他会卡在哪、担心什么」，需求就浮出来了（这和第三部分「走一遍用户旅程」是同一个方法）。另一半靠**真实的反馈**——用户会替你抓漏，会告诉你「这个地图怎么只画了一半」「这个链接跳过去打不开」。这些真实的不满，比任何调研都准。

**专用 Agent 做得深不深，取决于你对「用户到底要什么」理解得有多细。** 而这个理解，来自你愿不愿意真正站到用户的处境里、听不听得进他真实的抱怨——而不是坐在工位上，想当然地以为「用户要一段旅行文字」。
