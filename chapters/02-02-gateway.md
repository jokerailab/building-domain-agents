# 网关与 fallback 链

上一章把模型分了层。这一章解决三个工程问题：这些模型怎么**串成一条链**、主力挂了怎么**自动换**、以及在中国大陆的服务器上怎么保证这条链**连得上**——最后这一点，是通用架构书几乎不提、但做本土专用 Agent 绕不开的一课。

## fallback 链：挂一个，换下一个

最朴素的高可用，是一条降级链：主力挂了就试次选，次选挂了再试兜底。这个项目里是 `FallbackProvider`——它包着一串适配器，按顺序试：

```python
class FallbackProvider(LLMProvider):
    def chat(self, messages, tools=None):
        for idx, adapter in enumerate(self.chain):
            tries = 3 if idx == 0 else 1        # 链头多重试，后面的一次就过
            for t in range(tries):
                try:
                    return adapter.chat(messages, tools)
                except httpx.HTTPStatusError as e:
                    if e.response.status_code in (401, 402, 403, 408, 409, 429) or code >= 500:
                        time.sleep(1.5 * (t + 1))   # 退避后重试同一个，或换下一个
                        continue
                    raise
```

逻辑很直白：某个模型返回 401/403（key/权限问题）、429（限流）、5xx（服务端故障）——这些都是「这个模型这会儿指望不上」的信号，跳过它，换链里的下一个。

但这里有一个不那么直白、却很关键的设计：**链头（强模型）会多重试几次再放弃，后面的只试一次。**

为什么？因为强模型的 429/403 **大多是瞬时突发**——免费档的限流、并发抖动，退避个一两秒重试通常就过了。如果一撞到 429 就立刻掉到弱兜底模型，那就是上一章说的「智能污染」：明明强模型缓一秒就能用，你却因为一次瞬时抖动把整轮对话交给了驱动不了 function-calling 的弱模型，用户拿到一份变糙的行程。**多花一两秒退避重试，保住的是整轮的质量。** 这个「链头值得多等，尾部快速放弃」的不对称，是 fallback 链里一个容易被忽略的手艺。

## 区分「瞬时故障」和「真挂」

顺着上面这条线，还有一个更细的判断：**不是所有失败都该重试。**

看这个项目对错误的分类处理：

- **连接错误 / 5xx / 429** → 瞬时故障，退避重试（说不定重试就命中了健康的那个）；
- **ReadTimeout（模型在慢慢地不回你）** → **不重试**，直接抛给上层转成友好提示。

为什么 ReadTimeout 不重试？因为它不是「一下子失败」，而是「已经等了很久还没结果」。这时候重试，等于**再赔进去一个同样长的超时**——把用户的等待翻倍，而成功率并没有提高。区分「快速失败可以再赌一把」和「慢速失败别再赌」，是超时处理里的一个反直觉但重要的判断。

（真实数据佐证：这个项目里整份行程生成偶发要跑到两分钟，所以超时阈值给到了 150 秒——太短会把正常的长任务误杀成超时，太长会让真挂的请求干等。这个阈值是拿真实负载调出来的，不是拍脑袋。）

## 从「应用直连」到「统一网关」

早期，这些模型是应用自己直连各家厂商的——通义、Kimi、GLM，每家一个 base_url、一个 key，fallback 链写死在应用代码里。这能跑，但有个问题：**每次想调整模型池（换主力、加一家、调顺序、改路由），都要改应用代码、重新部署。** 运维和应用耦死了。

后来迁移到了**统一网关**（litellm 这类 OpenAI 兼容网关）。迁移之后，格局变了：

- **应用侧只认一个「组名」**——比如 `user-pool`。它不知道、也不需要知道这个组底下现在挂着哪几个模型、什么优先级、怎么降级容灾；
- **模型池、优先级、fallback、key、接入方式，全在网关一处维护。** 运维想把主力从 A 换成 B、加一条降级、调某个源的接入方式——在网关改，应用**一行不动、不用重新部署**。

还有一个细节很实用：网关在响应头里回一个字段（`x-litellm-model-id`），告诉应用**这一轮实际命中的是哪个模型**——因为应用只发了个组名 `user-pool`，不回报的话，应用根本不知道刚才到底跑的是链头还是降级到了尾部。上一章说的「模型徽章」，真实型号名就是从这个响应头来的。

**这是一个把「策略」和「机制」分开的经典架构收益**：应用负责「怎么用模型」（机制），网关负责「用哪些模型、什么顺序、怎么容灾」（策略）。策略是运维每天要动的，机制是稳定的——把它们解耦，两边都清爽。

## 可达性：本土 Agent 绕不开的硬约束

现在讲那个通用架构书不会讲的东西。

在境内部署、面向境内用户的 Agent，模型选型多了一个绕不开的维度——**它从你的部署环境能不能稳定连上。** 一个模型再强、再便宜，如果你的服务器发不出请求或收不到回复，它就等于不存在。现实是：**部分海外模型源在境内是访问受限的**——要接它们，得具备相应的额外网络条件（也就是常说的「科学上网」那类能力）。这件事本身我们不展开，这一章的重点也不在这里；重点是它对**架构选型**的影响。

那影响是什么？如果你把一个「要靠额外网络条件才连得上」的源放进 fallback 链，就等于拿产品的稳定性去赌一个不受你控制的网络条件。一旦某个降级路径命中了它，而那条件恰好不稳定，轻则请求超时、重则整个流程**冻死**（连接挂在那里，既不成功也不报错，用户对着一个冻结的页面一脸懵）——这个项目就实打实栽过：链降级到了一个境内不稳定可达的海外模型，把整趟排行程冻在半路。

这带来一个和「性能最优」完全不同的选型原则：**fallback 链里的每一个模型，都必须是从你的部署环境原生、稳定可达的。** 这个项目的链，是一串**在境内都能稳定连上**的模型——链头是国内直连的火山 GLM-5.2，撞限或失败就依次溢出到 ModelScope 的 DeepSeek、智谱 GLM-4.7-Flash，最后落到永久免费的 GLM-4-Flash 保险丝，每一档都在境内稳定可达。**那类「要靠额外网络条件才连得上」的海外模型，一律排除在链外**，哪怕它更聪明——一个会在某条降级路径上冻死整个流程的源，参数再好也是负债。

**「最强的模型」和「最合适的模型」在本土场景下经常不是同一个。** 做面向境内用户、部署在境内的专用 Agent，稳定可达不是一个可选的优化项，它是选型的**前置门槛**——一个连不稳的源，参数再漂亮，放进链里也是个定时炸弹。这一课，海外的 Agent 架构文章里几乎找不到，但它可能是你上线后遇到的第一个「为什么线上会莫名其妙全线冻死」的元凶。

## fallback 链的每一档，都得「还能开车」

选可达的模型只是及格线。还有一个更细、更容易漏的要求：**fallback 链里的每一档，不只要连得上，还要「接得住活」。**

具体到这个 Agent，「接得住活」意味着——**降级之后命中的那个模型，得还能驱动 function-calling**（连续调工具把行程排完）。上一部分讲过，有些模型虽然连得上、却带不动工具循环，一旦降级到它，流程照样卡死。一个连不上、或者连上了也驱动不了 Agent 的兜底档，是个假的安全网：它在链里，但你真降级到它时，它救不了你。

所以这条链是**按「能力」排的，不只是按「可达」排的**：链头是能稳定跑工具的强模型；往下每一档，直到最后那个永久免费的保险丝之前，放的都是**仍然能驱动 function-calling 的模型**（哪怕慢一些）。比如保险丝之上那一档，选的是一个还带得动 Agent 工具循环的免费模型，而不是直接跌到只会聊天、驱动不了工具的最弱兜底。这样即使一路降级，Agent 也还能把行程排完，只是慢一点。

正因为链本身就是这么排的，应用侧反而可以非常干净：

```python
def make_provider():
    if user_has_own_key:
        return user_provider                              # BYOK：走用户自己的模型
    return OpenAICompatAdapter("user-pool", gateway_url)  # 平台：只认一个组名，容灾全交给网关
```

应用只调一个 `user-pool` 组名，「挂了换谁、换的还能不能干活」全交给网关的链去兜。**当你的 fallback 链每一档都做到「可达且能干活」，你就不需要在应用里再堆一层层的安全带**——健壮性被设计进了链的组成里，而不是靠打补丁补出来的。

（顺带一提：用户自己配了 key 的 BYOK 路径，走的是自己的 provider，完全不经过网关和这些兜底逻辑——那是另一条独立的线，成本和额度都归用户，也不受平台这套可达性约束的限制，下一章讲并发和成本时会展开。）
