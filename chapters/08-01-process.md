# 做专用 Agent 的可复制流程

走完了从架构到产品的全程。这一部分，把这一路的东西，收成一套「**如果你要做一个新领域的专用 Agent，可以照着走**」的流程。

## 先泼盆冷水：它不是线性的

下面会把它列成「第一步、第二步」，但必须先说清楚——**真实的过程不是 1234 走一遍的瀑布。** 它更像这本书开篇讲的那个 agentic loop 本身：你会在某一步，发现前面的某个假设错了，然后回头改。列成步骤是为了讲清楚，别当成一条不回头的流水线。

带着这个前提，看下面这套流程。

## 第一步：走一遍用户旅程，列出「动作词典」

**别一上来就设计架构。** 工程师的本能是先搭框架，但做专用 Agent，第一件事应该是**理解领域**——真的去走一遍「用户从『想做这件事』到『做成』，中间要做哪些判断」。

每一个判断，就是一个候选工具（第三部分）。把这条真实旅程摊开，你的工具集——也就是这个 Agent 的能力边界——自然就浮现了。**这一步最该先做，却最常被跳过**，因为搭架构比理解领域「看起来更像在干活」。但顺序错了，后面全是返工。**领域理解，先于架构。**

## 第二步：搭通用骨架

有了动作，再搭承载它们的骨架：一个模型驱动的循环，加上工具、上下文、记忆、持续交互（第一部分）。

**这部分别自己发明，从成熟的范式（Claude Code 这类）去学、去抄。** 因为这是「通用」的部分——它换个领域基本不变，医疗、法律、旅行的 Agent，骨架都长一个样。在这上面自己发明轮子，是浪费。**把创造力省下来，留给下一步。**

## 第三步：把领域知识注入四个口

骨架搭好，开始灌「专业」——把领域的知识，按性质分别注入四个口（第四部分最后一章）：判断原则进 prompt、用法细则进工具描述、确定性规则进代码、可枚举的事实与判断进数据。

**这是「专业」开始显现的地方。** 同一套骨架，灌进旅行的知识就是旅行 Agent，灌进医疗的知识就是医疗 Agent。灌得准不准、全不全，决定了它有多「懂行」。

## 第四步：从第一天就守住真实性

**真实性不是后期补的，是从第一天就立的地基**（第三、第五部分）。

如果你的 Agent 要帮用户做真实决策，那么「禁止编造」这条铁律，要从架构第一天就在——而且不能只靠 prompt 恳求，要用架构让它**编不了**（把不能错的数据放到模型够不着的地方）；数据层要守那几条铁律（宁缺毋滥、参考的标参考、异常要标出来）。**真实性欠的债，越到后面越还不起**——因为一旦用户抓到一次编造，信任就崩了。

## 第五步：喂饱模型层、架好护栏

让这个 Agent 的「大脑」跑得稳（第二部分）：模型分层、fallback 链、可达性、并发闸门、弱模型护栏。

这一步的重心是「**稳」而不是「炫**」——不是追求用最强的模型，而是保证在真实的负载、真实的网络、真实的降级下，它**都能把活干完**、而且失败得体面可恢复。

## 第六步：用最轻的方式产品化

最后，把工具变成产品（第七部分）：账户、额度、商业化、拉新、平台化、设计系统。

贯穿这一步的判断是——**每一件事都选最轻的方案先把闭环跑通**（BYOK 先于支付、免费次数先于精确计量、邀请链接先于全民注册），验证了假设再上重的版本。**早做，但做轻。**

## 贯穿始终：实测验证

上面六步里，还有一件事不是「某一步」，而是**每一步都要做的**——实测验证（第六部分）。

不管是一段路线、一个工具、一次部署、一个页面，判断它「对不对」的标准，永远是「**我实际跑了一遍、观测到它是对的**」，而不是「代码看起来对」「AI 说没问题」。**这条贯穿始终——它不是流程里的一环，是流程里每一环的验收方式。**

## 这套流程的重心在哪

如果这套流程你只能记住一句话，记这句：**重心不在「搭架构」，在「理解领域」和「守真实性」。**

搭架构那部分（第二步），是可以抄的、会越来越标准化的；而「理解这个领域的用户到底要什么」「守住这个领域的真实性」，是没有现成答案、只能自己啃、也最决定产品能不能立住的地方。**很多人把大部分精力花在了那个可以抄的部分（架构），而在那个真正拉开差距的部分（领域理解 + 真实性）上投入不足——这是做专用 Agent 最常见的精力错配。**

---

> 这套流程里，第二步（搭骨架）是「通用、可迁移到任何领域」的，第三步之后（灌领域、守真实、做产品）是「这个领域独有、得自己啃」的。这两者到底怎么划分、又在哪里相乘？下一章正面拆这本书的核心命题——**通用架构 × 专业能力**，以及一个关于「壁垒在哪」的判断。
