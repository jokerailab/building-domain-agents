# 部署架构

前五部分讲的是「Agent 本身」——架构、模型、工具、业务、数据。从这一部分起，转向「怎么让它真正跑在服务器上、且跑得住」——工程与运维。先从部署讲起。

## 小系统的部署哲学：简单胜过「正确」

一提部署，很多人脑子里蹦出来的是 Kubernetes、微服务、多区域、自动伸缩。但对一个小团队的产品，**这些大多是负债，不是资产。**

这个项目的部署，朴素得几乎「不够先进」：**一台服务器、单文件式的代码同步、单 worker 的应用进程、一条隧道对外。** 没有容器编排，没有服务网格，没有多副本。

这不是将就，是**刻意的选择**。因为对一个小产品，**「简单到一个人能完全掌握」比「架构先进」重要得多。** 一套你自己都要查文档才能调试的复杂架构，在半夜出故障时是灾难；而一台服务器、一个进程、一条隧道，你闭着眼睛都知道哪里可能出问题。**部署架构的复杂度，要匹配团队的规模和产品的阶段——超前的架构，是在为不存在的规模，支付真实的复杂度成本。**

## 单 worker 的取舍：会话在内存里

这套简单架构里，有一个决定牵一发动全身——**应用跑单 worker（单进程）。**

为什么不多开几个进程扛并发？因为这个 Agent 的**会话状态是放在进程内存里的**。第一部分讲过，Agent 能暂停 / 恢复（用户答一个反问、几分钟后回来接着聊），这个「聊到一半」的状态，存在当前进程的内存里。如果开了多 worker，用户的第二条消息可能被负载均衡分到**另一个进程**——那个进程没有他这次会话的记忆，对话就断了。

要支持多 worker，就得给会话加**粘性路由**（保证同一用户始终落到同一进程）+ **共享会话存储**（把状态挪出进程内存）。这是一笔不小的复杂度。而当前的并发量，单 worker 加上第二部分的并发闸门，完全够用。

**所以单 worker 不是偷懒，是一个「现在还不需要付那笔复杂度」的清醒判断。** 知道「什么时候还不需要扩」，和知道「什么时候该扩」同样重要——过早地为扩展性买单，是小团队最常见的资源浪费。

## 单 worker 的坑：别阻塞事件循环

但单 worker 有一个必须警惕的坑，它能让整个站点瞬间卡死。

这个应用是异步框架（一个事件循环处理所有请求）。单 worker 下，**只要有一个同步的、阻塞的重活卡在事件循环里，整个站点的所有请求都会跟着冻住。** 这个项目踩过实实在在的坑：几个异步接口里，直接跑了同步的重活——一次几十秒的 LLM 调用、一次几十秒的批量数据补全——结果这几十秒里，**整个站点对所有用户都卡死了**。

修法是把这些重活**丢进线程池**执行，别让它们占着事件循环这条主干道；或者干脆把这类接口写成普通的同步函数，让框架自动放进线程池。判据很简单：**异步接口里，绝不能直接跑同步的重活。** 计算密集、或者会长时间等待的活，必须挪出事件循环。

这是单 worker + 异步框架组合下的头号陷阱——**它不报错，只是让全站一起卡几十秒**，而且很难在本地低负载时复现。知道这个坑存在，比事后排查它省事得多。

## 「慢」的真相：先定位，再优化

部署在一条隧道之后，你会遇到一个反直觉、但极其重要的性能真相。

用户反馈「慢」，你的第一反应可能是「去优化后端」。但实测这个项目：**后端在服务器本机直连，快到个位数毫秒；而外网访问测到的几百毫秒到一秒的延迟，几乎全部来自隧道和网络的往返（RTT），不是应用本身慢。**

这意味着——**如果你不先定位瓶颈在哪一层，就埋头优化后端，你会把大量精力花在一个根本不是瓶颈的地方。** 后端已经是毫秒级了，再快一倍用户也感觉不到，因为那点提升淹没在几百毫秒的网络往返里。真正的瓶颈在网络层，那就该在网络层想办法（CDN、就近接入），而不是继续抠后端代码。

**性能优化的第一步永远是定位，不是动手。** 「感觉慢」和「哪一层慢」是两回事——不先量出瓶颈在哪一层，优化就是在黑暗中挥拳。这一点重要到值得下一章专门展开。

## 部署要能一键，也要能验证

最后，部署本身要**简单到能一条命令搞定，且每次都自动验证**。这个项目的部署，就是一条脚本：把代码同步上去、重启服务，然后**自动去访问一下健康检查接口，确认它真的活着**。

「部署完自动验一下」这一步很关键——**部署脚本跑完不等于服务真的起来了。** 进程可能起来了但初始化失败、可能端口没监听上。加一个「部署完打一下健康检查」的自动验证，让「部署成功」这件事有一个客观的确认，而不是凭「脚本没报错」就以为好了。这也呼应后面「调试」一章的主题：**任何『应该成功了』的判断，都要有一个实际的验证兜底。**
