# 弱模型护栏

前两章尽力保证「跑起来的是好模型」——分层、fallback、可达性、并发闸门。但工程里没有百分百。总有那么些时候，请求还是落到了弱模型手里。这一章讲的，是**当降级真的发生时，怎么让它别把事情搞得一塌糊涂**。

先记住这一章的立场：**护栏的目标，不是让弱模型变聪明**（那做不到，上一章说过，那是能力下限）。护栏的目标是——**让弱模型搞砸的时候，搞砸得体面、可控、可恢复。**

## 弱模型不会说「我不行」

如果弱模型会老老实实报个错「这个任务我干不了」，事情反而简单。它最坑的地方在于——**它会一本正经地把事情搞砸，而且全程不报错。** 具体有三种典型的搞砸方式，这个项目都踩过：

1. **不收敛（死循环）。** 排行程要先「查真实游记核实路线」，弱模型会陷在这一步出不来——反复搜、大量返回 0 条、始终不往下走去排骨架。四五分钟过去，什么都没排出来，但它一点没停，还在搜。
2. **断流冻死（静默）。** 大请求在慢模型上超时，SSE 连接就挂在那里——既不成功、也不报错。前端一清理，用户对着一个冻结的页面，连个「失败了」都没有。
3. **吐畸形结构。** 让它输出结构化行程（`days` 是个数组、每天是个对象），弱模型会把整个数组 stringify 成一个字符串塞回来。下游代码一按数组去迭代，直接崩 500。

三种全都不报错。所以护栏的第一课是：**不能等模型「告诉」你它失败了，你得自己去检测、去兜。**

## 强制收敛：给循环装上刹车

针对「死循环」，护栏是一组**硬闸**——盯着循环的行为，到了某个阈值就强制它往前走。

最典型的是「查询硬闸」：核实类工具（搜索、研究、深读网页）的调用次数一旦超过阈值（比如 3 次），或者已经过了确认环节还在查——就**从工具列表里把这些查询工具摘掉，并回注一条强制提示「信息够了，现在去出结果」**。模型下一轮想再搜也没得搜了，只能往前走。类似的还有：反问用户超过三轮就强制截断转入设计；已经生成了一大段文字却还没提交，就回注强制提交。

这里有一个特别真实、特别隐蔽的坑，值得单独讲——**计数必须从「消息历史」里统计，不能用一个局部变量累加。**

为什么？因为这个 Agent 是可以暂停 / 恢复的（第一部分讲过）。如果你用一个函数内的局部变量数「搜了几次」，那么用户在 checkpoint 处暂停、再恢复时，驱动函数重新进入、这个局部变量**被重置为零**——于是恢复之后，计数从头开始，硬闸永远够不到阈值，死循环照样发生。改成从持久的消息历史里去数实际的工具调用次数，跨越暂停恢复才数得准。**护栏和「可恢复」这两个特性会互相咬**，这是只有真跑起来、真被用户中途打断过，才会暴露的坑。

## 卡位式护栏：关键步骤一步都不能漏

针对「跳步骤」（弱模型爱抄近道，跳过该做的核实），护栏是一组**必经关卡**。

比如自驾行程，「估算每段驾时」这一步不能省——不然弱模型可能排出「一天开 12 小时」这种离谱的腿。所以加了一道卡位：**要提交自驾行程、却没有核算过驾时 → 打回，先去估。** 同理，出报告前的 checkpoint 是必做的、critic 子 Agent 的核对是卡在 checkpoint 前的（第一部分讲过）。

这些卡位的共同逻辑是：**把「关键步骤」从「指望模型自觉去做」，变成「不做就过不去」。** 强模型或许能自觉走完全流程，但你不能赌每一次都是强模型在跑。卡位式护栏的价值，恰恰在模型变弱、开始抄近道的时候才显现——它是给「下限」兜底的，不是给「上限」锦上添花的。

## 防御畸形输出：像对待用户输入一样对待模型输出

针对「吐畸形结构」，护栏是**多层的输出校验**。

这个项目对「模型把 `days` 数组 stringify 成字符串」这类畸形，设了三道防线：

1. **入库前矫正**——有个 `_coerce_plan_shape`，专门把被 stringify 的数组 / 对象字段解析回来，丢掉非法元素；
2. **矫正不了就打回**——让模型重发一次；
3. **渲染层再兜一层**——就算前面漏了，模板渲染时也做防御，不让一个畸形字段崩掉整个报告页。

这背后是一条我认为做 Agent 必须刻进骨子里的原则：**对模型的输出，要像对用户的输入一样，默认不可信、层层校验。** 我们写 Web 后端时，绝不会相信前端传来的数据是干净的——同样，模型（尤其弱模型）返回的结构化数据，也绝不能假设它一定符合你要的格式。它会给你字符串化的数组、非对象的参数、缺字段的对象。**每一个「下游要按结构去用」的模型输出，都是一个校验点。** 少设一层，弱模型就能在那里给你崩一个 500。

## 失败也要体面且可恢复

最后，针对「断流冻死」，护栏的目标是把「静默失败」变成「体面失败」。

两件事：

**一、让失败被看见。** SSE 断流、模型超时这些场景，不能只是前端默默清理，得给用户一个明确的、友好的提示——「引擎有点忙，点这里重试（你的进度已经保留）」。**用户可以接受失败，但不能接受不明不白的失败。** 一句诚实的错误提示，比一个冻结的页面强一百倍。

**二、让失败可恢复。** 注意上面那句「进度已经保留」——这是关键。排到一半失败了，不该让用户从头再来。这个项目里，用户可以输入「继续出报告」，从上次的 checkpoint 处**续跑，而不是从零重排**。第一部分讲的可恢复架构（靠持久化的会话状态），在这里成了弱模型护栏的一部分：**因为状态是持久的，一次失败就只是「暂停」，而不是「清零」。**

这两件事合起来，是护栏的最高目标——**不追求不失败，追求「失败了也没关系」。** 一个偶尔失败、但每次失败都说得清、还能接着来的系统，用户是能容忍的；一个动不动就白屏、一失败就前功尽弃的系统，用户会直接走人。

## 回到那条铁律

绕了一圈，回到上一部分那条认知：**弱模型「变蠢」是能力下限，不是 bug，别当回归去修。**

护栏这一整套——强制收敛、卡位、输出校验、体面失败——没有一条是想「修好」弱模型的智力。它们全都是在承认「弱模型就是会搞砸」这个前提下，去**约束搞砸的方式和后果**：让死循环有刹车、让跳步骤过不去、让畸形输出崩不了、让失败可恢复。**你控制不了模型有多聪明，但你能控制它犯错时，你的系统有多稳。** 这就是护栏的全部意义，也是模型层最后、也最实在的一课。

---

> 分层、网关、闸门、护栏——这一切工程努力，都建立在同一个前提上：**「我们得自己提供模型」**。可如果这个前提本身该被挑战呢？护栏能兜住弱模型的下限，但兜不高它的上限。下一章是模型层的收尾，也是一个**策略转向**：当免费模型的天花板就是不够高时，与其在差模型上堆护栏、假装够用，不如换个思路——把「接入强模型」的门槛做顺滑，让用户带着自己的强模型进来。
