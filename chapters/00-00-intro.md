# 为什么要做「专用 Agent」

2025 年之后，Agent 这个词被用滥了。但如果把它拆开看，市面上真正跑通的 Agent 其实只有两类：**通用对话 Agent**（ChatGPT、Claude 这类什么都能聊）和 **Coding Agent**（Claude Code、Cursor 这类专注写代码）。

这两类都很强。但它们有一个共同的天花板——**它们不懂你的业务**。

## 通用 Agent 的边界在哪

你让 ChatGPT「帮我规划一条川西自驾环线」，它能给你一段看起来很专业的文字：稻城亚丁、色达、四姑娘山，配上「建议 7 天」。但你真要照着走，会立刻发现问题：

- 它不知道稻城亚丁回成都 817 公里、单天开不完，得在康定或雅安中途住一晚；
- 它给的机票价格是**编的**——它没有实时数据，只能凭印象说「大概两三千」；
- 它推荐的「越野车」可能是一个根本不存在的配置，或者压根不适合那条路;
- 它不会帮你把这些拼成一份**能直接出发**的东西：地图、每天住哪、开多久、花多少。

问题不在于模型不够聪明。问题在于**它没有业务的骨架**——没有真实的机票数据源、没有里程和驾时的计算、没有对「自驾」这件事本身的领域理解。通用 Agent 是一个博学但没有工具、没有数据、没有专业判断的通才。

## Coding Agent 为什么专用，值得学

Coding Agent 是第一个真正跑通的「专用 Agent」范式。Claude Code 之所以好用，不是因为模型更强，而是因为它把「写代码」这件事的**工作流**做进了架构里：

- 它有一套**工具**（读文件、改文件、跑命令、搜索代码），每个工具是一个明确的能力边界；
- 它是一个**模型驱动的循环**——模型决定下一步调哪个工具，看到结果再决定下一步，直到任务完成；
- 它管理**上下文**（哪些文件该读进来、长了怎么压缩）、有**记忆**（CLAUDE.md、跨会话）、能**持续交互**（问你、等你确认）。

这套架构是通用的。但它服务的领域——写代码——是专用的。**这正是专用 Agent 的模板：通用的 Agent 架构 × 专业领域的业务能力。**

## 那么，专用领域 Agent 是什么

如果把 Claude Code 的架构搬到「旅行」这个领域，会长成什么样？

这就是这套方法论要讲的东西。我做了一个叫 **野行 Y** 的产品——一个自驾行程规划 Agent，后来长成了一个自驾约伴社交平台。它的骨架完全对标 Claude Code：

| Claude Code | 野行 Y（自驾 Agent） |
|---|---|
| Agentic loop | 一样的循环，只是工具不同 |
| 读文件 / 改文件 / 跑命令 | 搜目的地 / 估里程驾时 / 查真实机酒 / 推车型 |
| CLAUDE.md 记忆 | 用户画像（去过哪、偏好什么） |
| 子 agent 并行 | research 并行核实多个景点 |
| 上下文压缩 | 一样的压缩，只是内容是行程不是代码 |

但**上层完全是旅行的业务**：什么是一条好的自驾环线、机票的真实性怎么保证、境内外的租车数据源差异、怎么把里程/驾时/住宿/预算拼成一份能出发的行程。这些是通用 Agent 给不了、Coding Agent 更给不了的。

## 这套方法论想讲什么

我的判断是：**这个时代，专用领域的 Agent 会越来越多。** 因为通用 Agent 和 Coding Agent 无法覆盖各行各业的专业场景——它们不够懂业务。医疗、法律、金融、旅行、教育……每一个专业领域，都值得有一个「既有 Agent 架构、又懂本行业务」的专用 Agent。

而做这样一个 Agent，需要同时驾驭两件事：

1. **通用的 Agent 架构**——loop、工具、上下文、记忆、模型、并发、运维。这部分可以从 Claude Code 这样的成熟范式学。
2. **专业的业务能力**——领域知识怎么建模、工具怎么定义、数据真实性怎么保证、用户到底要什么。这部分没有现成答案，得自己啃。

这套方法论，就是把我做野行 Y 的整个过程——从架构、模型、工具、记忆，到业务建模、数据真实性、工程运维、产品化——一章一章拆开写下来。**每一章既讲技术，也讲业务；既有通用的方法，也有真实的坑。**

不是教程，是一个产品创作者的实战复盘，和一套「如何做专用领域 Agent」的可复制方法。
