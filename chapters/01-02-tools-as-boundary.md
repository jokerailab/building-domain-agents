# 工具即能力边界

上一章说，Agent 是一个「循环」——模型每一轮决定下一步做什么。但「下一步」不是凭空的：模型能做的每一件事，都是一个**工具**。循环决定了「有得选」，工具决定了「能选什么」。

一句话：**工具就是 Agent 能对真实世界做的全部动作。** 模型再聪明，它对世界的影响也只能穿过工具这道门。所以设计专用 Agent，工具的定义方式，直接决定了它的能力上限。

## 一个工具是什么：中性的四元组

先看这个项目里工具的定义。它非常小：

```python
@dataclass
class Tool:
    name: str                       # 模型看到的名字
    description: str                # 模型什么时候、怎么用它
    parameters: dict                # JSON schema，模型要填的参数
    run: Callable[[dict], str]      # 真正干活的函数，返回字符串
```

关键在这段注释（`app/agent/tools.py` 开头写着）：

> 循环不关心一个工具背后是原生 Python 函数、一次 MCP 调用、一个命令行子进程，还是一个服务端工具。

这一点很重要。`search_destinations` 是查本地数据库，`web_search` 是联网抓网页，`recommend_vehicle` 是查自建车型库，未来还可能挂一个 MCP。但对循环来说，它们**长得一模一样**：一个名字、一段说明、一组参数、一个返回字符串的函数。循环只做一件事——把工具列表交给模型，模型挑一个、填好参数，循环执行 `run`，把返回塞回上下文。

这个「中性四元组」是整套架构的解耦点。**能力可以无限往上加，循环一行都不用改。** 这是从 Claude Code 学来的最值钱的一个抽象：工具是能力的最小单位，也是唯一单位。

## 工具集，就是这个领域的「动作词典」

Claude Code 的工具是：读文件、改文件、跑命令、搜代码。这四个动作，构成了「写代码」这件事的全部原子操作。

那「排一趟自驾」的原子操作是什么？这就是做专用 Agent 时，你必须自己回答的问题。野行 Y 的工具集是这样长出来的：

| 阶段 | 工具 | 对应的领域动作 |
|---|---|---|
| 挑地方 | `search_destinations` | 从目的地库里按地貌/大区筛候选 |
| 核实 | `research` / `web_search` / `read_source` | 并行核实多个点 / 单点补查 / 深读某页正文 |
| 算路 | `estimate_leg` | 估一段的里程和驾时（防一天开 12 小时） |
| 选车 | `recommend_vehicle` | 按路况×人数×预算给选车指南 |
| 问与确认 | `ask_user` / `checkpoint` | 收集缺的需求 / 主线骨架给用户拍板 |
| 让用户挑 | `fetch_options` / `choose` | 查真实机酒车选项 / 原样透传给用户选 |
| 出结果 | `submit_plan` / `edit_plan` | 生成完整行程 / 局部改某几天 |

**做专用 Agent 的第一步，不是画流程图，是列这张动作词典。** 流程交给循环（上一章讲过），你要做的是把领域拆成一组正交的、模型能理解的动作。列这张表的过程，本身就是在逼你把「排行程」这件模糊的事，拆成一个个说得清、做得到、可核实的动作。

注意这里有个和 Coding Agent 的差异：旅行 Agent 的工具里，有一类 Claude Code 没有的——**和用户来回的工具**（`ask_user` / `checkpoint` / `choose`）。写代码大多时候可以闷头干，但排行程绕不开「几天、几个人、什么车、这版行不行」这些只有用户知道的信息。所以「交互」在这个领域不是附加功能，它本身就是几个一等公民的工具。这条线会在本部分「持续交互」一章展开。

## 描述即文档：把规则下沉进工具，别堆进 system prompt

这是我在这个项目里做的一次重要重构，也是最反直觉的一点。

一开始，我把所有规则都写在一个巨大的 system prompt 里——「搜索时要怎样」「查了要写进 notes」「多个点要并行不要一个个查」……写着写着，system prompt 涨到 70 行。它成了一坨没人敢动的东西：改一条规则要在长文里找半天，而且**每一轮对话都要把这 70 行重新喂给模型**，又贵又容易被稀释。

后来我把这些规则**下沉进了各个工具自己的 description**。举两个真实的例子：

`web_search` 的说明里直接写着：

> 查到的头牌景点写进对应那天 highlights，路况/封路结论写进 notes（**查了不写＝没查**）。

`research` 的说明里写着：

> **一批要核实的就用这个，别一个个 web_search**。

重构之后，system prompt 从 70 行瘦到约 25 行——它只留「你是谁、整体流程、几条铁律」，**每个工具怎么用的细则，跟着工具走**。

为什么这样更好？三个理由：

1. **就近原则**。关于 `web_search` 的规则，只有模型在考虑要不要调 `web_search` 时才需要看到。写在工具描述里，模型选这个工具的那一刻它就在眼前；写在全局 prompt 里，它是一段永远常驻、大部分时候用不上的噪音。
2. **成本**。system prompt 每一轮都在花钱（还没算它对模型注意力的稀释）。工具描述只在工具列表里出现一次，边际更划算。
3. **内聚**。工具的「代码」和它的「用法说明」放在一起，改一个自然想到改另一个。这跟好代码把函数和它的 docstring 放一起是同一个道理。

**一个好工具是自文档化的。** 模型不需要读一篇手册才会用它，看工具本身的名字、描述、参数就够了。这也正是 Claude Code 的工具给人的感觉——你几乎从不需要告诉它「用 Read 工具时要注意什么」，工具自己说清楚了。

## 参数不只是入参，是在「教模型翻译」

工具的 `parameters`（JSON schema）常被当成技术细节，其实它是设计里信息密度最高的地方。**每个参数的 description，都是一条「把用户的自然语言，翻译成结构化输入」的指令。**

看 `recommend_vehicle` 这个工具的参数，就明白了。它不只收一个「地点」，而是收了一整套：

- `terrain`：路况描述，「川藏318 高原 烂路涉水」还是「城市周边 高速铺装」——判越野需求；
- `trip_type`：出行性质，「带老人」「带小孩」「露营」「商务接待」——影响舒适/空间/要不要回避硬派越野；
- `budget`：预算档，经济/适中/不限；
- `preference`：选车偏好，性价比/舒适/豪华/人多/四驱/新能源/纯电/增程/省油……
- `brand`：品牌倾向；
- `named`：用户点名问的具体车型（「坦克500 能进藏吗」「汉兰达和普拉多选哪个」）。

这套参数的用意是：**用户说的每一句关于车的诉求，都有一个格子接住它。** 用户说「一家六口，带老人，想舒服点，别太贵」，模型的任务就变成把这句话拆进 `party_size=6` / `trip_type=带老人,家庭` / `preference=舒适` / `budget=适中`——然后背后的车型库（一个真实存在的判断层，「数据护城河」一章会讲）按这些维度算出该推赛那还是坦克。

我在工具描述里专门写了一句引导模型「把用户所有诉求翻译成参数」，因为**参数填得越全，推荐越准**。参数 schema 在这里不是约束，是一张「问卷」——它替你规定了「要把一个模糊需求看成哪几个维度」。设计这张问卷，就是在做领域建模。

## 返回结构：一次返回，喂给两个读者

工具的 `run` 返回一个字符串（通常是 JSON）。这个返回值有两个去处，它们的需求完全不同：

- **喂回模型的上下文**——模型要拿它做下一步判断，越精炼越好，无关字段是负担；
- **给用户看的活动流**——「搜到 5 个航班，最低 ￥820」这种一行人话，让用户知道 Agent 在干什么。

同一个返回值服务两个读者，是专用 Agent 里一个反复出现的张力。这个项目用两个函数把它们分开：`_slim_tool_result` 把结果瘦身后喂模型，`_summarize_result` 把结果压成一行人话给 UI。这背后是完整的「上下文工程」——它值得单开一章，就是下一章。

这里先记住结论：**工具的返回不是「打印出来给人看」，而是「结构化地喂给模型」。** 它是模型下一轮判断的原料，所以它的结构、精简程度、错误怎么表达，都是要设计的。

还有一个特殊的返回边界值得点一句：`submit_plan` 是**唯一**能生成行程报告的工具。模型可以搜、可以问、可以估、可以选，但「把这一切拍成一份能出发的行程」只有这一个出口。这相当于给循环设了一个「提交点」——类似写代码里的 commit。有这样一个明确的边界工具，循环就有了清晰的终点：模型知道「什么时候算干完了」，护栏也知道「在哪一步核对约束」（上一章的 checkpoint 就卡在它前面）。

## 删掉死工具，也是一种设计

重构时我做的一件小事：删掉了 `build_itinerary`、`search_flights` 这两个**定义了却没注册进工具表**的函数。

它们是早期的残留，代码还在，但循环根本不会把它们交给模型。留着它们没有功能上的害处，但它有一个隐蔽的坏处：**工具表是「这个 Agent 能做什么」的唯一真相来源。** 表里有一个模型永远调不到的工具，或者代码里有一个看起来像工具、实际是死码的函数，都是在对下一个读代码的人（包括半年后的你自己）撒谎。

所以在专用 Agent 里，我把工具表当成一份需要保持诚实的清单来维护：注册进去的，就是模型真能用的；不用了的，删干净。**能力边界要清清楚楚，一个不多，一个不少。**
