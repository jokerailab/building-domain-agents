# 模型是体验的地板

前面四章——分层、网关与可达性、并发闸门、弱模型护栏——本质上都在做同一件事：**把「免费模型」榨到「能用」。** 用尽了各种工程手段：多档 fallback、可达性筛选、排队限流、给弱模型架护栏。

但做到这一步，会撞上一个用工程手段绕不过去的事实。这一章正面讲它，也是模型层的收尾——它是一个**策略转向**。

## 模型能力，是垂类 Agent 体验的地板

先说这个事实：**排一趟行程，是「几十轮工具调用 + 流式 + 长任务」的场景。** 这个场景，会把模型的每一个弱点都放大：

- **慢** —— 单轮慢几秒不显眼，但一趟几十轮累加起来，就是好几分钟，甚至撞上超时「出不来」；
- **限流** —— 免费档的速率限制，在密集的多轮调用里频繁触发；
- **不稳** —— 某一轮换了个更弱的模型接手，工具调用就崩、流程就断；
- **飘** —— 弱模型把地名定到千里之外，一条线拉出几千公里的鬼影。

**差模型在这个场景里，不是「差一点」，是「体验塌了」。** 一个能聊天的模型，未必能稳稳地跑完一趟排行程。**模型能力，是这类 Agent 体验的地板**——地板矮，上面盖再多护栏、做再多补偿，也撑不起一个好体验。这跟单轮问答的产品很不一样：那里模型弱一点，答案糙一点，还能用;而在长任务的 Agent 里，模型弱一档，整个任务可能就完不成。

## 「能用但体验差」，比「暂时不能用」更伤

这里有一个反直觉、但我认为极其重要的产品判断。

**第一印象只有一次。** 如果一个新用户来排第一趟行程，你用一个差模型给他一个「慢、飘、半天排不出来」的初体验——他会当场给这个产品判死刑：「这东西不行」，然后走掉，而且带着这个印象再也不回来。

对比之下，「暂时不能用」（比如：需要先接入你自己的模型 Key）反而**没有留下一个坏印象**。它只是一道门槛，不是一次失望。

**「能用但体验差」比「暂时不能用」更伤产品**——因为前者消耗掉了你仅有的一次第一印象，还留下了「这产品很糟」的负面记忆；后者只是把用户挡在门外，门里的东西还是好的。**拿差模型去稀释核心体验，是拿仅有的一次第一印象去赌，而且是笔烂赌。**

## 于是：BYOK 优先，免费只作可控内测

想通这一点，策略就清晰了——**用 Agent 排线，必须接入你自己的模型 Key（BYOK）。**

- **你的 Agent，你的模型，越强越先进越好。** 我们甚至不替用户推荐具体型号——因为强模型池还没被充分评测过，与其给一个可能过时的推荐，不如把选择权和「越强越好」的原则交给用户。
- **免费模型收敛为内测通道。** 它不再对所有普通用户开放，而是收进一个**白名单**——只放给特定的种子用户，并且明确告诉他们「这是测试版、免费模型能力有限」。免费不再是「默认体验」，而是「一个可控的、说清楚了预期的内测入口」。
- **不依赖强模型的功能，照常开放。** 逛约伴广场、看别人的行程、管理自己的车队、看通知、提心愿——这些都不需要强模型，无 Key 也照常用。

**门槛只设在真正需要强模型的那个动作（排线）上，别一刀切。** 一个新用户能进来、能逛、能看，只有当他要「让 AI 帮我排一趟」时，才遇到「接入你的 Key」这道门。这样门槛拦住的，恰恰只是那个「对体验最敏感、最不能用差模型糊弄」的核心动作。

## 门槛要设在对的地方，还要做到极顺滑

设门槛，最怕设成一堵墙。所以这道 BYOK 的门槛，**必须做到极顺滑**——顺滑到用户「带着自己的强模型进来」几乎无感：

- 进设计台，就**主动引导**（而不是等他撞墙才提示）；
- 引导是**内联的弹窗**，不跳走；选一个平台，就**自动预填**好接入地址；
- 贴上 Key、填上模型名，**一键启用**；
- 启用之后，**自动接着把他刚才想说的那句话发出去**——他不用重来一遍。

**门槛的作用不是「拦人」，是「把人平滑地导到『用强模型』这条对的路上」。** 一道设计得好的门槛，用户跨过去几乎不觉得是道坎；一道设计得差的门槛，会在他最有兴趣的那一刻把他劝退。同样是「必须接 Key」，顺滑与否，决定了它是增长的漏斗，还是流失的闸刀。

## 用数据决定「用哪个模型」，不拍脑袋

BYOK 优先，不代表「不管模型了」。恰恰相反，配套建了一套**模型评测子系统**，作为「用哪个模型」的**唯一数据依据**：

- **持续、多维的探针** —— 可靠性、稳定性、速度（首字延迟 + 吞吐合成的「趟耗时」）、工具调用成功率、成本，各维度持续打分；
- **生产的真实收敛率** —— 真实排程里，谁能稳稳收敛出报告，谁的分就高；
- 综合成一个分，驱动选型。

这么做，是因为选模型最忌讳「凭一次手测拍脑袋」——一会儿看延迟、一会儿看吞吐，反复横跳。**把选型建立在持续、统一、可追溯的数据上，让它越用越准，而不是每次凭感觉。** 而这套评测的第一个硬发现，本身就支持了 BYOK 优先的决定：**当前能拿到的免费池里，没有一个是「又快、又稳、又不限流」的**——数据摆在那儿，与其在这样的池子上继续堆工程，不如把强模型的选择权交给用户。

## 为什么（现在）不做「SaaS 封装」

还有一条看起来更「产品化」的路：平台把强模型封装好、开箱即用，用户什么都不用配，平台替他承担模型成本。

这条路体验最好，但它需要**一整套基建**——支付、计费、额度核算、成本风控。对早期的产品，这套基建太重了，重到会拖慢你验证产品本身。**BYOK 用最轻的方式，跳过了这整套基建**：用户自带成本，你不碰钱、不做计费，同时把「当前最强的模型能力」直接交到了用户手里。等产品真的验证了、规模真的起来了，再考虑要不要为「开箱即用」建那套重基建也不迟。（这正是第七部分「每一步选最轻的方式先跑通」的又一次应用。）

## 收束这一部分

回看整个模型层：分层、网关、可达性、并发闸门、弱模型护栏——这些是你在「必须自己提供模型」的前提下，能做的所有工程努力，都值得做（尤其是为那个内测白名单池）。但它们有个共同的天花板：**再好的工程，也拔不高模型能力这块地板。**

所以垂类 Agent 早期，最诚实的做法是：**别在差模型上堆护栏、假装体验够好；而是把「接入强模型」的门槛做到最顺滑，让用户带着自己的好模型进来；免费只作为一个说清了预期的、可控的内测通道存在。** 地板不够高时，不要粉刷地板——换一块用户自己的好地板。
