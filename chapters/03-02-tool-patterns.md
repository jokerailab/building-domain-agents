# 工具设计模式

上一章把旅行的工具集拆了出来。这一章往下一层：**这些工具具体怎么设计。** 会发现一件让人安心的事——工具虽多，但它们不是各不相同的，而是反复落在**五种模式**上。认得这五种模式，你设计任何专业领域的工具，都有章可循。

这五种是：**搜索、估算、推荐、核实、选择。** 下面一个个拆。

## 模式一：搜索（从一个库里筛出候选）

代表工具：`search_destinations`（从目的地库里按地貌 / 大区筛）。

**它干的事**：接一个模糊的查询，从一个**你自己掌握的数据集**里，返回一组带结构化属性的候选。`search_destinations` 返回的每个目的地，都带着地貌、海拔、独特度评分——不是一段描述文字，是**能让模型继续做判断的结构化字段**。

设计要点：

- **返回结构化属性，不是散文。** 模型要拿这些属性去比较、去筛。返回「稻城亚丁，海拔 4700m，独特度 9」，比返回「稻城亚丁是个很美的地方」有用一百倍。
- **控制返回量。** 默认返回十几条、最多几十条，别把整个库倒给模型——那是上下文灾难（第一部分讲过）。
- **它和「核实」是两回事。** 搜索是「从我的库里挑」，核实是「去外面的世界确认」。别糊在一起。

## 模式二：估算（把一个量算出来）

代表工具：`estimate_leg`（估一段的里程和驾时）。

**它干的事**：接两个点，算出一个**数值**——多少公里、开多久。看它的实现，朴素得几乎可爱：

```python
km = haversine(a, b)          # 两点直线距离
road_km = km * 1.35           # 乘一个绕路系数
hours = road_km / 55.0        # 除以山路有效时速
```

直线距离 × 1.35 绕路系数、除以每小时 55 公里的山路时速。这不精确——它自己在返回里都老实标着「仅供每日行程可行性粗判」。但它**足够好**，因为它的用途不是导航，是**让模型判断「这一天是不是排得太赶」**。一天估出来要开 11 小时，模型就知道该把这天拆成两天——这个判断，用一个粗估就够了，不需要接一个精确路由 API。

设计要点：

- **精度要匹配用途，不是越准越好。** 估算类工具最容易过度设计——一上来就想接高德、Mapbox 算精确路线。但如果模型只是用它做「可行性粗判」，一个 haversine × 系数就够了，省钱、省依赖、还不会因为外部 API 挂了而卡住。真正精确的里程，留到最后出报告时再用真实路线数据补。
- **返回里带上「这个数怎么来的」。** `estimate_leg` 的返回里写明了系数和时速假设。这让模型（和看代码的人）知道这个数的可信度边界，不会拿一个粗估当精确值用。
- **诚实标注不确定性。** 这是估算类工具的品德——算不准的地方，明说算不准。

## 模式三：推荐（把需求翻译成一个判断）

代表工具：`recommend_vehicle`（按路况 × 人数 × 预算 × 偏好选车）。

**它干的事**：接一堆用户诉求，输出一个**带理由的判断**。这是五种模式里最「重」的一种，因为它内部藏着真正的领域知识。

比如选车，它内部有一个 `terrain_need`——把「川藏318 高原 烂路涉水」这样的路况描述，翻译成一个 1-5 的越野需求分：

```python
def terrain_need(*texts) -> (int, str):
    # 匹配路况关键词，取命中规则里最高的越野需求，附一句依据
    # "烂路涉水" → 5 分；"城市周边高速" → 1 分；识别不出 → 默认 3
```

然后再叠加人数、预算档、出行性质（带老人就回避颠簸的硬派越野）、品牌倾向，算出一个推荐分和理由。**这一整套「需求 → 判断」的映射，就是这个领域的专业知识被编码进了工具。**

设计要点：

- **推荐类工具的核心，是那套「判断逻辑」，不是数据本身。** 谁都能列一张车型表，但「一家六口带老人走川西该选什么」这个判断，是稀缺的。这套判断逻辑值钱到可以单独成为护城河——下一章「数据护城河」专讲。
- **输出必须带理由。** 只给结果不给理由的推荐，模型没法向用户解释，用户也不敢信。「推荐赛那，因为七座够坐、舒适性好、带老人不颠」——理由让推荐可信、可复述。
- **别输出你不确信的东西。** 这条通向下一章的主题——真实性。推荐可以有判断，但判断要基于真实参数，参数不确定就留空，绝不编造。

## 模式四：核实（去外部世界确认真假）

代表工具：`research` / `web_search` / `read_source`。

**它干的事**：搜索是「从我的库里挑」，核实是「去外面的世界确认」。它专门对付一件事——**防止模型编造。** 模型脑子里「觉得」某个垭口 10 月能过、某个景点存在，但它可能记错、可能过时、可能就是幻觉。核实工具让它把「觉得」变成「查过」。

设计要点：

- **核实工具的产出，要能变成行程里的一条依据。** `web_search` 的描述里明写着「查到的路况结论写进 notes，**查了不写＝没查**」——核实不是走个过场，结论要落进行程、让用户看得到。
- **它是防幻觉的第一道闸。** 核实这一类工具存在的全部理由，就是「模型的记忆不可信，得对外求证」。这条原则重要到值得单开一章讲，就是下一章。
- **按代价分档**（上一章讲过）：单点 `web_search`、批量 `research`、深读 `read_source`，让模型按需选最省的。

## 模式五：选择（把真实选项交给用户拍板）

代表工具：`fetch_options` + `choose`。

**它干的事**：有些决定不该 Agent 替用户做——选哪班航班、住哪家酒店。这类工具**查出真实的候选，原样交给用户挑**。

它的设计要点，第一部分「持续交互」的防幻觉闭环已经讲透了，这里只强调一个模式层面的要点：**选择类工具，模型全程不能碰关键数据。** 航班号、价格从查询到落地一直待在代码里，模型只负责「发起查询」和「透传展示文本」。因为一旦让模型经手价格，它就可能「顺手」改一个、编一个。**选择类工具的铁律，是把不能错的数据挡在模型够不着的地方。**

## 五种模式，一张对照表

把这五种模式并排放，设计要点一目了然：

| 模式 | 代表工具 | 输入 | 输出 | 最容易踩的坑 |
|---|---|---|---|---|
| 搜索 | search_destinations | 模糊查询 | 结构化候选 | 返回散文而非属性、和核实混淆 |
| 估算 | estimate_leg | 两个点 / 参数 | 一个数值 | 过度追求精度、不标注不确定性 |
| 推荐 | recommend_vehicle | 一堆需求 | 带理由的判断 | 有结果没理由、编造不确信的参数 |
| 核实 | research / web_search | 要确认的问题 | 事实结论 | 查了不落进行程、当过场走 |
| 选择 | fetch_options + choose | 查询条件 | 真实候选给用户挑 | 让模型经手关键数据 |

**认得这五种模式，设计新工具就不再是从零发明。** 拿到一个新的领域动作，先问它属于哪一种：是从库里筛（搜索）、算个量（估算）、给个判断（推荐）、对外求证（核实）、还是让用户拍板（选择）？归了类，那一类的设计要点和坑，就直接套用。这套模式不只适用于旅行——任何专业领域的 Agent，工具大抵都逃不出这五种。

---

> 这一章里，「真实性」这个词反复冒出来：核实是为了防编造、推荐不能编参数、选择不能让模型碰价格。这不是巧合——**真实性是专用 Agent（尤其是要给用户做真实决策的）的生命线。** 下一章把它单独拎出来讲透：为什么真实性优先于一切，以及为了它，这个项目做了哪些近乎偏执的取舍。
