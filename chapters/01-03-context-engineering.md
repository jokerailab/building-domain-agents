# 上下文工程

模型没有记性。它不「记得」上一轮发生过什么——每一轮，它「知道的一切」，就是这一次喂进上下文窗口的那些字。窗口里有的，它才看得见；窗口里没有的，对它就不存在。

所以**上下文工程 = 经营这个窗口**：每一轮，往里放什么、把什么挡在外面、太长了在哪压缩、除了对话还要注入什么。做专用 Agent，这是最容易被忽视、又最影响体感的一门手艺。忽视它，你的 Agent 会又贵、又慢、还越聊越蠢。

## 窗口是稀缺资源，也是唯一的真相

先建立两个认识，它们是这一章所有判断的出发点：

1. **窗口是模型唯一的真相来源。** 模型不会「回忆」，它只会读窗口。放错了东西，它就基于错的东西幻觉；漏放了关键约束，它就当那个约束不存在。
2. **窗口是稀缺的。** 它有上限，而且越长越贵、越慢，还会稀释注意力——真正重要的那句话，淹没在一堆无关内容里，模型反而看不见了。

一个自驾行程排下来，模型可能调十几次工具：搜目的地、并行核实五六个垭口、估七八段驾时、查机酒……**如果每个工具的原始返回都原样堆进上下文，窗口会以肉眼可见的速度膨胀。** 尤其是 `web_search`——它的返回又长（每条结果一大段 snippet）、又多（一趟排线要搜很多次），是上下文膨胀的头号元凶。

不管它，后果是复利式的：第 10 次搜索时，前 9 次的完整网页摘要都还压在窗口里，模型每一轮都要重读一遍它早就用过、再也用不上的东西。又贵，又慢，又蠢。

## 一个工具结果，两个读者：就地瘦身

上一章埋了个钩子：工具的返回值有两个读者——**模型**（要精炼）和**用户的活动流**（要人话）。上下文工程的第一招，就是把这两个读者分开伺候。

喂给模型的那一份，做**瘦身**。这个项目里是 `_slim_tool_result`（在 `app/agent/loop.py`）：

```python
def _slim_tool_result(name: str, result: str) -> str:
    """喂给模型上下文的工具结果瘦身版（UI 活动流仍用完整版）。"""
    # 只有 web_search 需要瘦身，其它工具本就紧凑或数据关键
    if not result or name != "web_search" or len(result) <= 500:
        return result
    d = json.loads(result)
    # 每条结果只留 标题(截 70) + 摘要(截 150)，丢掉 url 和多余字段
    slim = [{"title": r["title"][:70], "snippet": r["snippet"][:150]}
            for r in d["results"][:6]]
    return json.dumps({"query": d["query"], "count": d["count"], "results": slim})
```

逻辑很朴素：一条搜索结果，模型真正用得上的只有标题和一小段摘要——用来判断「这个景点是不是真的、这条路通不通」。url、多余字段、超长的正文，模型写行程时根本用不到。所以进模型之前，把它们砍掉，只留标题和截短的摘要。

实测下来，单次搜索的 token 降了一半多。因为是复利的，整趟排线省下来的量非常可观。

而给用户看的那一份不动——UI 的活动流仍然拿完整数据，配一个 `_summarize_result` 压成一行人话：「搜到 5 条：稻城亚丁自驾攻略；折多山垭口路况」。**同一个工具返回，喂模型的瘦、给人看的全，各取所需。**

这里有个容易做错的地方，值得强调：**瘦身不是无脑地「把所有结果都截断」。** 注意上面那个 `name != "web_search"` 的判断——只有 `web_search` 被瘦身，别的工具原样放行。为什么？因为像 `fetch_options` / `choose` 这类工具的返回是**要原样透传给用户去选的真实机酒选项**，价格、航班号、订票链接一个字段都不能少，砍了就出错。`estimate_leg` 的返回本就紧凑，没必要动。

**判断哪个字段模型用得到、哪个工具的数据碰不得——这才是上下文工程的核心，不是「压缩」这个动作本身。** 压缩谁都会写，知道压什么、不压什么，是领域判断。

## 在工具边界上就地蒸馏

瘦身是「砍字段」，还有一招更进一步：**在信息进窗口之前，就把它蒸馏成结论。**

`research` 工具（并行核实多个点，下一章「子 Agent」细讲）是最好的例子。模型传进来五个要核实的问题（五个垭口的路况），它并行搜完，拿到的是五坨原始搜索摘要。如果直接塞进上下文，又是五大段。

但 `research` 没这么干。它在返回之前，先用一次 LLM 调用，把每个主题的搜索摘要**压成一句事实结论**：

> 折多山：10 月通常可通行，遇强降雪偶有临时管制，建议出发前查路况。

进到主循环上下文里的，是五句这样的结论，而不是五坨摘要。**压缩发生在信息进窗口「之前」，在工具边界上就地完成，而不是等窗口涨起来之后再回头收拾。**

这是两种不同的思路：`_slim_tool_result` 是「进门时搜身」，`research` 的蒸馏是「在门外就把话说成一句」。真实系统里两种都要有——前者兜底防膨胀，后者从源头上让高噪音的工具（搜索类）只往窗口里放干货。

## 不是所有上下文都来自工具：注入什么

窗口里的内容，除了对话历史和工具返回，还有一部分是你**主动注入**的。这部分同样是上下文工程，而且往往决定了 Agent 「像不像个懂行的人」。这个项目注入三样东西：

**一、运行时注入当前日期。** system prompt 里不写死日期，而是每次运行时把「今天是 2026-XX-XX」拼进去。听起来是小事，但少了它，模型会拿训练时的默认日期给你算机票、给你填一个已经过去的出发日。一句注入，根治一类幻觉。

**二、首轮注入用户画像。** 老用户开一趟新行程，系统会调 `build_user_profile(uid)`（在 `trips_social`）把他的偏好、去过哪些地方拼进第一轮的 system 里。于是 Agent 一上来就「记得」这个人——「你上次去过川西，这次给你换个方向」。这是**短期上下文**和**长期记忆**的接缝：记忆系统负责跨会话把用户存下来（下一章的主题），上下文工程负责在合适的时机、以合适的量，把它注入回窗口。注意是「合适的量」——不是把用户所有历史全倒进去，那又会膨胀，而是提炼成一小段画像。

**三、system prompt 本身。** 这是最大、最常驻的一块注入。上一章讲过，我把它从 70 行瘦到约 25 行——把细则下沉进了各个工具的描述。这件事在「工具」那一章是「自文档化」，在这一章它是另一个名字：**system prompt 是每一轮都要付费的常驻上下文，它的每一行都在花每一轮的钱、占每一轮的注意力。** 能挪到工具描述里按需出现的，就别让它常驻。两章讲的是同一个动作的两面。

## 让上下文扛得住中断

最后一点，是真实系统才会碰到的：上下文会被**打断**。

用户排到一半合上电脑，第二天回来接着聊；或者一次长任务中途暂停、稍后恢复。这时候上下文不能「断片」——尤其是进度。这个项目里加了一个 `convStage`，让进度跨 pause-resume 不回退：恢复之后,Agent 知道自己排到了哪一步，而不是从头再来一遍，或者更糟——以为没做过、把已经确认的东西又问一遍。

这提醒我们：**上下文不是一次性喂完就完事的,它是一个要跨越暂停、恢复、甚至崩溃重启而保持连贯的状态。** 经营好它，Agent 才经得起真实世界里断断续续的使用节奏，而不只是在一次干净的 demo 里跑通。

> 这一章讲的是**短期**上下文——一次会话内窗口里的经营。但用户的偏好、去过的地方、这个人是谁，不该每次从零开始。窗口装不下、又需要跨会话记住的东西，交给**记忆系统**。下一章讲 Agent 怎么「记得你」，以及怎么防止它记岔了、把没发生的事当成真的。
