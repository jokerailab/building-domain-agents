# 从业务动作到工具

第一部分讲了工具的**机制**（工具即中性四元组、自文档化、参数即翻译）。第二部分讲了驱动工具的**大脑**（模型层）。现在回到工具，但换一个视角——**专业领域的视角**：旅行这件事，到底需要哪些工具？

这不是一个技术问题，是一个**领域建模**问题。而它恰恰是做专用 Agent 最核心、也最难的一步。通用 Agent 和专用 Agent 的分水岭，就在这里：**通用 Agent 什么工具都没有（或只有搜索），专用 Agent 有一组为这个领域量身定制的动作。**

## 工具集 = 把领域任务拆成正交的动作

「帮我排一趟川西自驾」是一个模糊的、复合的任务。模型没法一步做完它。你要做的，是把这个大任务，拆成一组**原子的、正交的、可核实的动作**——每个动作对应一个工具。

野行 Y 拆出来的这组动作是这样的：

| 领域动作 | 工具 | 它回答什么问题 |
|---|---|---|
| 挑地方 | `search_destinations` | 「哪些地方符合『高原/草原/环线』？」 |
| 核实真实性 | `research` / `web_search` / `read_source` | 「这个垭口 10 月通不通？这个景点真的存在吗？」 |
| 算路 | `estimate_leg` | 「这一段开多久、多远，一天能到吗？」 |
| 选车 | `recommend_vehicle` | 「这条路况、这些人，该开什么车？」 |
| 查真实机酒 | `fetch_options` | 「这天从 A 到 B 有哪些真实航班？」 |
| 收集需求 / 确认 / 选择 | `ask_user` / `checkpoint` / `choose` | 「几天几个人？这版行不行？你选哪个？」 |
| 出结果 | `submit_plan` / `edit_plan` | 「把这一切拼成一份能出发的行程 / 改其中几天」 |

拆解的时候，有三个标准我会反复拿来检验：

**一、正交**——两个工具的职责不该重叠。`search_destinations`（挑地方）和 `web_search`（核实）是两件事，不要糊成一个「万能搜索」。职责一重叠，模型就会犯迷糊：这事该调哪个？

**二、可核实**——每个工具的产出，最好能落到一个客观事实上。`estimate_leg` 算出「523 公里 / 9.5 小时」，这是能验证的；而一个「帮我看看这条线好不好」的工具，产出是主观的，没法核实，也就没法建护栏。

**三、模型能理解**——工具的名字、参数、返回，模型看一眼就懂什么时候用、怎么用（第一部分「自文档化」）。一个模型搞不清何时该调的工具，等于没有。

## 动作从哪来：走一遍真实的用户旅程

那这组动作是怎么想出来的？不是凭空设计的，是**走一遍真实用户从「想出去玩」到「能出发」的全过程**，看这中间他（或一个帮他的老司机）实际要做哪些判断。

一个人要自驾川西，真实要经历的是：先定个大概方向（去哪片、什么地貌）→ 上网查查这条线靠不靠谱、什么季节去、有没有封路 → 掰着手指算每天开多久、住哪 → 琢磨开什么车合适 → 订机票酒店 → 出发。**每一个「要做的判断」，就是一个候选工具。** 把这条真实旅程摊开，工具集自然就浮现了。

这也解释了为什么专用 Agent 的工具集，是**通用 Agent 给不了的**。通用 Agent 不知道「自驾要先算驾时防止一天开太远」，因为它没有走过这条领域旅程、没有把「算驾时」识别成一个必要动作。**它有一个博学的大脑，但没有这套领域动作的手脚。** 你做专用 Agent 的价值，一大半就在于把这套「手脚」设计出来、装上去。

## 一个动作，可能拆成好几个工具

有时候一个「动作」不够原子，要再拆。「核实」就是个例子——它在野行 Y 里其实是三个工具：

- `web_search`：核实**单个**点，临时补查一下；
- `research`：一次并行核实**一批**点（多个垭口、多个景点）；
- `read_source`：某个网页的摘要不够了，深读**那一页的正文**。

为什么一个「核实」要拆三个？因为它们的**代价和场景不同**：单点补查用 `web_search` 就够；一批要核实的用 `research` 并行（快，第一部分讲过）；只有当摘要说不清某个具体细节（景区几点关门、某段路况），才值得用 `read_source` 去深读整页。**把一个动作按「代价 / 精度」拆成几档，让模型按需选最省的那档**——这既是工具设计，也是成本设计。

反过来也成立：不该拆的别硬拆。`submit_plan` 内部其实自动聚合了机票、酒店、车型、路线一大堆事，但它对模型只暴露成**一个**「出报告」的动作。因为对模型来说，「把设计好的行程拼成报告」是一个完整的意图，没必要让它去手动编排底下那十几步。**工具的粒度，要匹配模型的「意图颗粒度」，不是匹配你代码的模块划分。** 这是个很容易做错的地方——工程师习惯按代码结构拆工具，但工具是给模型用的，该按「模型脑子里的一个个意图」来拆。

## 边界之外：什么不该做成工具

最后一个判断，同样重要：**不是所有事都该做成工具。**

有些领域知识，更适合放进 **prompt**（比如「自驾环线要闭合」「带老人要控驾时」这类判断原则）；有些数据，更适合直接查**数据库 / 静态文件**，在工具内部用掉，而不暴露成一个模型要显式调的工具。工具是给模型「做决定」用的动作，如果一件事模型不需要决定「做不做、怎么做」，它就不该是工具——它可能是工具内部的一个实现细节，或者一条 prompt 规则。

判据还是那句：**工具是「模型要自主决策的动作」。** 模型不需要决策的，往下沉进实现；模型需要遵守的原则，写进 prompt；模型需要主动去做、且要根据情况选择的，才做成工具。把这三者分清楚，工具集才会干净——不多不少，正好是模型需要亲自操作的那些动作。
