# 从工具到产品

前面六部分，把一个能用的 Agent 造出来了。但**能用的工具，还不是一个产品。** 产品要能认得用户是谁（账户）、能自己活下去（商业化）、能让人进得来（拉新）。这一章讲从工具到产品的那几步——而会看到，每一步真正的关键，都是同一句话：**选最轻的方式，先把闭环跑通。**

## 工具和产品，差在哪

工具是「能用」，产品是「能自己活」。中间差的，是账户、额度、拉新这些「**非功能**」的东西——它们不直接让 Agent 更聪明，但它们决定这个 Agent 能不能变成一门能持续下去的生意。

对一个产品创作者，这些「非功能」的东西有个特点：**最容易被拖到最后，又最决定生死。** 你会本能地想先把功能做爽，账户、收费「以后再说」。但「以后」往往是产品有了真实用户、想收钱、想拉人时的手忙脚乱。这一章想说的是——**这些步该早做，但要用最轻的方式做。**

## 商业化：BYOK 先行，把最重的门槛后置

商业化的第一步，直觉是「接支付」。但支付是**最重的门槛**——要资质、要合规、要对接、要处理退款和风控。把它当第一步，你会在还没验证「有没有人愿意为这个产品付出代价」之前，就陷进一堆合规工作里。

更轻的跑通方式是 **BYOK（自带 Key）**：用户的免费额度用完后，填自己的模型 API Key，就能继续用。**无支付、无资质，当天就能上。**

BYOK 巧妙在哪？它用一种「零合规成本」的方式，验证了一个关键假设——**有没有人愿意为『继续用』付出代价。** 只不过这个代价暂时是「他自己的 Key（他自己承担的模型成本）」，而不是「打给你的钱」。如果连填个自己的 Key 都没人愿意，那说明产品的价值还不够，急着接支付也没用；如果有人愿意 BYOK，你就验证了付费意愿的存在，再上真正的付费订阅、按量计费就有底气了。

**BYOK 是商业化的一个绝佳的第一步：它把「最重的支付合规」后置，先用一个当天能上的方案，验证最核心的那个假设。**

## 额度：别一上来就做精确计量

有了「免费 + 付费能力」的分层，下一个诱惑是把额度做**精确**——按 token 计量、每次调用扣多少、钱包、流水、定价……这套东西很完整，也很重。

这个项目一度设计过完整的 credit 计量，最后**砍掉了**，换成一个朴素得多的模型：**免费次数。** 每人免费用 N 次，用完就引导去 BYOK。一个整数计数器，就够把「免费边界 → 引导升级」这个闭环跑通了。

为什么？因为在产品早期，你根本不需要精确到 token 的计量——你需要的只是**一条清晰的免费边界**（用完了、该引导了）。精确计量解决的是「精细定价」的问题，而精细定价是一个你还远没到的阶段的问题。**用一个整数计数器，先把『免费到付费的过渡』这件事立起来；等你真的要按用量精细定价了，再上重的计量系统。** 过早做精确计量，是在为一个还不存在的定价策略，支付真实的复杂度。

## 鉴权：也是选最轻的

拉新的第一步是让人能进来，也就是鉴权。这里同样有「重」和「轻」的选择：微信登录要企业认证、审核、年费；短信验证要签名审核、要花钱。这些对一个还在 beta 的产品，都是过重的门槛。

更轻的方式是**邀请链接**：点一个链接、填个昵称，就进来了，不走微信、不走短信。对一个「先拉一批种子用户试水」的阶段，这完全够用——你要的是「让对的人快速进来」，不是「支持全国人民扫码注册」。

**拉新的第一步，同样是用最轻的方式先让人进得来。** 重的注册体系（微信、手机号），等产品验证了、要规模化拉新了，再上。

## 贯穿的判断：先跑通，再加重

把这三件事放一起看，会发现一条清晰的主线：

| 这件事 | 重的方案（后置） | 轻的方案（先跑通） |
|---|---|---|
| 商业化 | 支付订阅、按量计费 | BYOK 自带 Key |
| 额度 | token 精确计量 + 钱包流水 | 免费次数计数器 |
| 拉新 | 微信 / 手机号注册 | 邀请链接 |

**不是那些「重」的方案不好，是它们不该是第一步。** 产品化的每一步，都先用最轻的方案把闭环跑通、验证假设，再决定要不要为它上更重的版本。**过早上重方案，是在一个还没被验证的假设上，押很大的工程成本**——万一假设不成立（没人愿意付费、没人愿意来），那套重系统就白建了。

## 但「非功能」要尽早，别真拖到最后

要澄清一个容易被误读的点：**「选最轻」不等于「拖到最后再做」。**

账户、数据归属、额度这些，一旦产品有了真实用户，就**绕不开**——用户的行程要归属到他名下、免费得有个边界、Key 得按人隔离。而且它们越晚接，改造成本越大：比如一个「匿名就能跑」的产品，事后改成「登录才能跑」，会牵动一大堆已有的路径和数据。

**所以正确的姿势是：早点用最轻的方式，把这层地基打上。** 早做、但做轻——用邀请链接而不是微信登录、用免费次数而不是 credit、用 BYOK 而不是支付。这比「先纯做功能、以后再补账户体系」要省事得多。**产品化的地基要趁早打，只是打的时候，用最省力的那个版本。**
