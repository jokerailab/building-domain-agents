# 持续交互

前面五章讲的是 Agent「自己怎么想」。这一章讲它**怎么跟用户来回**——这是专用 Agent 和「一次性问答」最不一样的地方，也是第一部分架构地基的最后一块。

一次性问答是：你说一段，它答一段，结束。但排一趟行程不是这样——它得先弄清你几天几个人、排到一半给你看主线让你拍板、查到真实航班让你挑一班、出完报告你还想「D3 改成 2 晚」。**全程有来有往，像 Claude Code 排改代码时那样：它自主推进，但在关键处停下来、等你、然后接着干。**

## 默认一把梭，该停才停

先说一个容易走偏的地方：持续交互，**不等于问个没完**。

一个每一步都停下来问你「接下来查酒店可以吗」的 Agent，比一个闷头干完的还烦。所以野行 Y 的默认行为是**一把梭**——信息够了就直接排、直接出报告，快。只有两种情况才停：

1. **信息真的缺**——不知道从哪出发、几个人、哪天走，没法开工；
2. **决定值得你拍板**——主线骨架排好了，出报告前让你确认一眼；或者查到几班真实航班，你想自己挑而不是用系统默认。

**「持续交互」的精髓不是多问，是停在对的地方。** 该自主的地方别打扰用户，该让用户做主的地方别自作主张。这个分寸，是交互设计里最难也最值钱的判断。

## 三种停顿原语

野行 Y 把「停下来」抽象成三个工具，各管一类停顿：

| 工具 | 什么时候停 | 停下来干什么 | 用户能做什么 |
|---|---|---|---|
| `ask_user` | 信息缺 | 收集需求 | 点选 / 填日期 / 填数字 / 打字 |
| `checkpoint` | 出报告前 | 摆出已做的决定（主线骨架） | 确认 / 调整 / 你看着办 |
| `choose` | 有真实候选 | 让用户挑机票/酒店/车 | 单选 / 多选 / 用默认 |

三个工具，对应三种不同的「人在环」：`ask_user` 是**要输入**，`checkpoint` 是**要拍板**，`choose` 是**要选择**。它们在上一章「工具即能力边界」里已经作为一等公民出现过——旅行 Agent 有一整类 Coding Agent 没有的「和用户来回」的工具，这里把它们讲透。

有几个刻意的设计值得点出来：

- `ask_user` **最多问两轮**（第三轮系统强制截断），而且每题尽量给可点选的选项 + 带理由的默认值——「建议 7 天，梅里 + 雨崩至少 5 天」。**别让用户打字，别挤牙膏。** 一个好的提问，是把用户的决策成本压到「点一下」。
- `checkpoint` 是 `submit_plan` **前的必做一步**。它摆出逐日主线（每天带里程），让用户在「出完整报告」这个重动作之前，先花五秒确认方向对不对。上一章的 critic 子 Agent 就卡在它前面——**先让 critic 挡掉硬伤，再让用户看一眼没硬伤的主线**。
- `choose` 让用户挑的，必须是**真实**的候选（下面「防幻觉闭环」细讲）。

## 一条 pause/resume 管道：resume 靠的是 id，不是工具名

这三种停顿，工程上怎么实现？这里有一个不显然、但特别漂亮的设计，值得单独讲。

一个 Agent 循环跑到 `checkpoint`，要停下来等用户——可用户可能五秒后点确认，也可能合上电脑明天再来。**循环不能干等着**。所以它的机制是：跑到暂停型工具，就把「我停在这个工具调用上」这件事记下来，然后**把循环挂起、连接断开**；等用户答复来了，再**从那个点恢复**。

恢复靠什么定位「停在哪」？答案是**工具调用的 id**（`tool_use id`），**不是工具的名字**。模型每次调工具都会带一个唯一 id，暂停时记的是这个 id；用户答复时带着同一个 id 回来，循环就知道「哦，是那次 `checkpoint` 的答复」，把答复塞回去、接着跑。

这个「靠 id 不靠名字」的选择，带来一个极其省事的结果：**新增一种停顿类型，几乎不用改恢复逻辑。** `checkpoint` 和 `choose` 是后加的，但它们复用了 `ask_user` 早就铺好的那条恢复管道——因为管道只认 id。想加第四种、第五种停顿？照样白嫖这条管道。

这是一条很通用的架构经验：**当你有多个「变体」要走同一套流程，让流程认一个中性的标识（id），而不是认具体类型。** 认类型，每加一个变体都要在流程里加一个分支；认 id，流程一次写好，变体随便加。这跟第二章「工具即中性四元组」是同一种思路的延续——**在正确的地方做抽象，能力就能无痛地长。**

## 边做边说：让等待不枯燥

停下来是一种交互，**不停下来的时候**也是。

排一趟行程，模型在后台可能要搜、要算、要核实十几次，这中间用户在干嘛？盯着一个转圈图标。体验很差。所以 `SYSTEM_PROMPT` 里有一条「边做边说」的规则：调工具的过程中，用**简短、口语、老司机顾问的口吻**说一两句正在想什么——

> 「这条线 10 月垭口可能有雪，我查下封路」
> 「卡兹别克值得留 2 晚，我按深度排」

这些话是流式（streaming）实时吐给用户的。于是等待从「盯着转圈」变成「听一个懂行的人一边干活一边念叨」。规则里还特意约束：**每句一两行，别吐长文、别列整份行程**——行程交给最后的报告页，聊天流里只给「过程感」。

这看起来是个体验细节，但它其实是「持续交互」的另一半：交互不只是「停下来问」，也是「**干活时让用户看得见你在干活**」。一个沉默三十秒然后突然甩出报告的 Agent，和一个全程有声有色的 Agent，用户对它的信任度完全不同——哪怕它们最后的结果一模一样。

## choose 的防幻觉闭环

`choose` 让用户挑机票、酒店、车。这里藏着一个必须用架构来堵的坑：**绝不能让模型编选项。**

如果让模型「生成三个航班选项」给用户挑，它会编出以假乱真的航班号和价格——用户选了一个「东航 MU5432，￥820」，结果这班飞机根本不存在。这是灾难。

所以 `choose` 的候选走一条严格的闭环：

```
模型调 fetch_options(kind=flight)     # 模型只发起"去查"
   → 代码真查 Travelpayouts           # 真实数据源，拿到真航班
   → 真实字段缓存起来（按 key）        # 价格/航班号存在代码这边
   → 只把展示字段 label/sub 给模型     # 模型看到"东航 MU5432 08:20"，但不碰它、也改不了
用户 picked 某个 key
   → 代码从缓存取真实字段注入行程      # 用户选的就是真实那一条
```

关键在于**模型全程不碰真实数据**。它只做两件事：发起查询、把查回来的展示文本原样透传给用户看。价格、航班号这些「不能错」的字段，从查询到落地一直待在代码里，模型没有机会篡改或编造。用户选了哪个 key，代码就从缓存里把那一条真实数据注入行程——**所见即所得，选的就是真的**。

前端也守着同一条线：`choose` 卡片原样渲染后端给的 label/sub，**不去解析、不去重算**真实字段——因为多一层解析，就多一个引入偏差的机会。

**防幻觉不能只靠在 prompt 里写「不要编」，得靠架构让它「编不了」。** 把不能错的数据挡在模型够不着的地方，是专用 Agent（尤其涉及价格、时刻这类硬信息的领域）的一条通用防线。这条线会在第五部分「数据与真实性」里反复出现，`choose` 是它第一次登场。

## 局部编辑：改一天，不动其余；改错了，能撤销

报告出来了，用户说「D3 改成住 2 晚」「换个民宿」「加一天去 X」。

第一版我做错了：续聊改行程走的是 `submit_plan` **整份重排**。结果用户只想动 D3，整条线连没让动的天都被重新洗了一遍——酒店换了、顺序变了。这不是「调整」，这是「重来」。Claude Code 改代码时不会因为你说「改这一行」就重写整个文件，行程也不该。

所以加了 `edit_plan`：模型**只列要改的那几天**（`replace` / `insert_after` / `delete`），`_apply_day_edits` 按天号应用，**未提及的天原样保留**。这里有个真实的坑：模型给的天号常常是字符串 `"3"` 或 `"D3"`，得兼容；更重要的是——**`replace` 找不到对应的天时，必须跳过，绝不能当成「插入」**，否则会莫名其妙多出一天（测试里川西曾被改成 8 天）。局部编辑的「局部」二字，全靠这些边界守得住。

配套还有一个 `undo`：每次编辑前把旧版整份存进 `_prev`，报告页给一个「撤销上次调整」的按钮。**能局部改，还能反悔**——这才是让用户敢在一份已经不错的行程上继续动手的底气。

## 打断即接上，和一个交互上限

最后两个细节，都关于「用户不按你的节奏来」。

**打断即接上。** Agent 正在跑，用户又想起一句「对了，避开收费路」。老实现是：忙的时候输入框禁用，或者把这句话吞掉。新实现是——把这句话**排进队列**，显示「⏳ 已排队…」，等这一轮到了安全的边界就停下、把排队的话接上继续。这就是 Claude Code 式的 interrupt + insert：**你随时能插话，它不丢、也不乱**。（注意：排队只在跑动中处理，正停在 `ask_user`/`checkpoint` 等你答复的时候不处理——那时候它本来就在等你。）

**一个交互上限。** `ask_user` + `checkpoint` + `choose` 合计有个次数上限（约 6 次），到顶就优雅地直接出报告。为什么要设这个？因为**交互是手段，不是目的**。用户来是要一份行程，不是来做问卷的。一个会没完没了确认的 Agent，和一个从不确认的 Agent，是同一种失败的两面。设一个上限，等于给「持续交互」这件事本身也加了一道护栏——**懂得适可而止地停，和懂得在关键处停，同样重要**。
