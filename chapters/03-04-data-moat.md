# 数据护城河

上一章说，真实性靠真实数据源。但数据源本身是有层次的——有的是**公开的、谁都拿得到的**，有的是**你得自己建、别人抄不走的**。这一章讲后者：怎么给专用 Agent 造一条数据护城河。

先给结论：**最值钱的数据，不是能搜到的规格，是没人结构化过的判断。** 拿旅行来说——「坦克 500 的离地间隙是多少」这种规格，网上到处都是；但「一家六口带老人走川西烂路，该开什么车」这种判断，**任何一个现成的数据库、API 里都不存在。** 这个空白，就是护城河的位置。

## 三层数据：能搜的、能买的、得自己建的

做选车这个功能时，我先去找现成的数据。调研下来，数据分成很清楚的三层：

| 层 | 例子 | 谁都能拿到吗 |
|---|---|---|
| **通用规格** | 车的座位、马力、能源 | 能，公开数据一堆 |
| **专业硬参数** | 离地间隙、涉水深度、接近离去角、四驱形式 | 难，几乎没有干净的免费聚合，只散落在少数参数页里 |
| **领域判断** | 「这车适合什么路况、什么人开、别选它的场景」 | **不存在**——没有任何现成数据把它结构化过 |

第一层没有壁垒——谁都能拿到的东西，构不成竞争力。第二层有点门槛（数据散、没有干净的**免费**聚合，得自己去抓；花钱买商业数据能省这道力气，但那是另一套成本账），本质上还是「客观事实」，别人肯花力气也能凑齐。**真正的护城河在第三层：把车和「自驾场景」之间的匹配判断，做成结构化数据。** 这一层花钱也买不到——因为它不是「事实」，是「经验」，全世界都没有谁把它整理成一张能查询的表。

## 事实 + 判断：一个车型条目长什么样

野行 Y 自建了一个车型库（`data/vehicles.yaml`，约 193 个条目、覆盖几百款车）。它的关键，不在「有多少车」，在**每个条目的结构**——它把「事实」和「判断」缝在了一起：

```yaml
- model: 坦克300
  # ——事实层（可从公开参数页抓取校准）——
  body: 硬派越野SUV
  seats: 5
  drive: 分时四驱
  clearance: 224        # 离地间隙 mm
  wading: 700           # 涉水深度 mm
  approach: 33          # 接近角
  # ——判断层（自建，护城河）——
  offroad: 4            # 越野力 1-5（自驾视角评分）
  comfort: 3
  space: 3
  good_for: [川西/西北环线, 轻中度越野, 搓板路炮弹坑]
  avoid: [长途高速要舒适, 一家老小求安逸]
  verdict: 硬派入门标杆，越野够用、公路稍颠
  need: [四驱, SUV, 越野]
```

上半截是**事实**——离地间隙、涉水深度这些客观参数，可以从公开的汽车参数页抓取、校准。这半截有价值，但不独特。

下半截是**判断**——`offroad/comfort/space` 的自驾视角评分、`good_for`（适合什么）、`avoid`（什么场景别选）、`verdict`（一句话结论）、`need`（匹配标签）。**这半截是护城河。** 它不是从哪抓来的，是「一个懂自驾的人」对每款车的判断，被结构化地记了下来。「坦克 300 适合搓板路炮弹坑、但长途高速求舒适就别选」——这句话，你在任何汽车网站的参数表里都找不到。

**规格是事实，判断是资产。** 事实谁抓都一样，判断是你注入的领域经验——它才是别人抄不走的东西。

## 判断层不是一张表，是一套「推荐逻辑」

光有每款车的判断标签还不够。护城河的另一半，是把这些标签**用起来**的那套逻辑——一个能把「用户的模糊需求」翻译成「排序推荐 + 理由」的判断引擎。

这个项目里是 `vehicles.recommend()`。它接一堆用户诉求，输出带理由的推荐：

```python
def recommend(abroad, party, prefs, *terrain, budget, trip_type, named, brand):
    need, why = terrain_need(*terrain)      # "烂路涉水" → 越野需求 5 分
    for v in lib:
        score, reasons = fit_reason(v, need, prefs, party, budget, brand)
        # 出行性质回避：带老人小孩 → 硬派越野/皮卡软降权（颠、不便）
        if v["body"] in avoid_bodies:
            score -= 3
    # 按 路况 × 人数 × 偏好 × 预算 × 出行性质 × 品牌 综合排序
    ...
```

它内部把好几个维度揉在一起打分：路况需求（`terrain_need` 把「川藏318 烂路涉水」翻译成 1-5 的越野分）、人数（六口人自动并入「人多」）、预算档、出行性质（带老人就给硬派越野降权，因为颠）、品牌倾向。最后输出的不只是「推荐坦克 500」，而是「**推荐它，因为四驱够硬、离地间隙够、适合你说的烂路**」——带着理由。它甚至能做「点名评估」：用户问「坦克 500 能进藏吗 / 汉兰达和普拉多选哪个」，它对每款车给出针对这趟行程的适配判断。

**这套「需求 → 判断」的映射逻辑，和那张判断标签表，合起来才是完整的护城河。** 表是「原料」，逻辑是「把原料变成决策的工艺」。两者都得自建，缺一个都不成壁垒——光有表没逻辑，是本查不动的字典；光有逻辑没表，是台没原料的机器。

## 护城河怎么建、怎么养

自建判断层，有几个我踩下来觉得关键的点：

**一、别贪全，只做「决策相关」的。** 这个库明确不做成汽车参数网站——不追求每款车的全部参数，只留「自驾选车真正关心的」那几个关键项 + 判断结论。**护城河的宽度，不在数据多全，在判断多准。** 做全是数据网站的活，做准才是你的活。

**二、事实层可以自动化，判断层必须人工注入。** 客观参数（离地间隙这些）可以写个抓取脚本从公开参数页补全、校准——这部分是体力活，能自动化。但判断层（评分、适合 / 慎选、一句话结论）**是人的经验，自动不来**，得一款款注入。**把「能自动的」和「只能人工的」分开，你的人力就花在刀刃（判断）上，而不是浪费在抄参数上。**

**三、宁缺毋滥，拿不准的留空。** 参数只填确信的公开规格，拿不准的宁可留空、等抓取补全——这是上一章「真实性」在数据层的延续。**护城河也建立在真实之上，一个塞满了猜测的判断库，是负资产。**

**四、它是增量资产，越养越深。** 车型库可以持续扩充、判断可以持续校准。**护城河的特点就是「时间友好」**——今天建一点、明天补一点，别人想追，得把你积累的判断从头攒一遍。这种靠时间和领域经验堆出来的东西，恰恰是纯技术（模型、代码）最难复制的。因为模型能力大家很快会拉平，代码可以照抄，但**你对这个领域一款款车、一条条路攒下来的判断，抄不走。**

## 对专用 Agent 意味着什么

模型会越来越强、越来越便宜，这意味着「用了什么模型」很快不再是壁垒。那专用 Agent 的壁垒在哪？

在**那些模型不知道、也搜不到的领域判断**——而你把它结构化下来了。

1. **找到你领域里的第三层数据**——那些「不是事实、是经验」，没人结构化过的判断；
2. **把它做成「事实 + 判断」的结构化资产**，事实自动化、判断人工注入；
3. **配一套把它用起来的推荐逻辑**，让判断能被查询、被组合、被解释；
4. **持续养它**——护城河是时间的朋友。

**通用能力（模型、架构）会被抹平，专业判断（数据护城河）才是你真正独有的东西。** 这也正是第一部分那个命题的另一面：专用 Agent = 通用架构 × 专业能力——通用架构是入场券，专业判断才是护城河。
