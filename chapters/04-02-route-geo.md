# 目的地与路线

上一章说，用户要一份「能出发」的行程，而它的骨架是路线。这一章讲路线——但会发现，路线是这个领域里**最不讲情面**的一块：文学修辞能糊弄，地理糊弄不了。一条线排得漂不漂亮是主观的，但「这天开不开得到」「这个点在不在这儿」是客观的，错了就是错了。

## 一条好的自驾线：环线要真闭合

自驾的招牌玩法是**环线**——从一个城市出发，绕一大圈，最后回到起点（或同一个交通枢纽），一路不走回头路。规划一条漂亮的环线，是这个领域的硬功夫。

但「环线」不是一个文学词，它是一个**几何约束**：终点必须回到起点或枢纽。模型很容易排出一条「看起来是环线」、实际上终点甩在离出发地几百公里外的线——读起来顺，真去走，最后一天发现回不了家。所以「环线是否真闭合」，是出行程前必须核对的一条硬约束（第一部分的 critic 子 Agent 就把它列为核对项之一）。

**领域里很多概念，在文字上是修辞，在落地时是约束。** 「环线」「顺路」「不折返」——这些词模型都会说，但只有把它们变成可核对的几何 / 里程条件，它们才真正成立。做专用 Agent，一大半工作就是把这类「听起来对」的说法，钉成「可以验证对不对」的约束。

## 驾时和里程：地理设定的硬上限

比环线更硬的约束，是**驾时**。

一个人一天能开多久，是有上限的。稻城到成都八百多公里，单天开不完——你必须在中途（康定或雅安）安排一晚。这不是偏好问题，是物理问题。**地理设定了「一天能走多远」的硬上限，行程必须在这个上限内排。**

排不好会怎样？模型（尤其弱模型）会排出「一天开 12 小时」这种腿——文字上就是一行「D3：A 到 B」，读不出问题，真去开，天黑了还在山路上。所以这里需要两道东西：

- **粗估自检**：排线时用一个粗估（第三部分的 `estimate_leg`）快速算每段大概开多久，超了就把这天拆成两天。这个估不用精确，够判断「是不是太赶」就行。
- **真实里程兜底**：出最终报告时，用真实路线数据算准确里程，别拿粗估糊弄用户。
- **硬守卫**：单日里程 / 驾时超过一个明显不合理的阈值，直接判定「这天排错了、该拆」。

**在旅行这个领域，驾时是比价格、比景点都更底层的约束**——景点可以换、预算可以调，但「一天开不到」是没法商量的。把它作为一条硬护栏，是自驾 Agent 和「只会拼景点」的旅游 Agent 的分水岭。

## 地图上的真实：坐标是隐形的地基

路线要画在地图上。而地图，是这个领域**最容易翻车的隐形地基**——因为它依赖一件看起来简单、实则脆弱的事：**把地名变成坐标**（geocode）。

坐标错一个，整张地图就歪。这个项目在坐标上踩过的坑，几乎每一个都值得记下来：

**一、洋地名查不到，那天就悄悄消失了。** 境外落脚点靠地名查坐标，但很多地方的常用名和官方名不一样——比如格鲁吉亚的 Kazbegi，官方名其实是 Stepantsminda。查不到坐标，代码就静默跳过这天——结果地图上这天**没有点、没有路线**，用户看到的是一张缺了好几天的残图。这引出一条验证的教训：**验境外地图，不能只看「坐标有没有落在目标国家」，必须逐天查「每一天的路线都画出来了没有」**（路线段数要等于自驾天数、日签要齐）。「抽样看着对」和「每天都对」是两回事。

**二、一个错点，能带偏整张图。** 某个景点的地名 geocode 错了，坐标飘到千里之外（一个江南的景点被定位到几百公里外的海边），如果不管，它会把整天的落脚点、甚至整张地图的范围都拽偏。解法是一道**离群守卫**：一个点如果离所有主要落脚点都远得离谱（比如超过 120 公里），判定它是错点，直接丢弃，别让它污染地图。

**三、不开车的行程，地图规则完全不同。** 如果用户是坐飞机高铁玩（不自驾），地图就不该画驾驶折线、不该把出发和中转的大城市当成游玩节点、地图范围只该框住实际游玩的地方——否则「苏州玩三天」的地图会被硬拽到把上海也框进去。**同一个「画地图」的动作，自驾和非自驾是两套逻辑。**

这些坑的共同点是：**它们在代码里不报错，在地图上才现形。** 一个 geocode 失败静静返回空、一个错误坐标安静地画了个点——程序都觉得自己没错。只有人打开地图，才看出「这天怎么没了」「这个点怎么在海里」。**地理类功能的验证，必须靠眼睛看渲染结果，不能靠代码不报错。**

## 境内和境外，是两套世界

上面的坑还指向一个更大的结构：**境内和境外，在地理这件事上是两套完全不同的世界。**

- **数据源不同**：境内用一套在国内又全又准又自洽的地图服务，境外用另一套开放地图数据——两套不能混用，混用就会出现「洋地名被定位到中国」这种诡异错误。
- **地名规则不同**：境内地名规整，境外常用名 / 官方名 / 中英文名各一套，查坐标的命中率天差地别。
- **路况规则不同**：境内境外连「能不能自驾」的前提都不一样。

**做一个覆盖境内外的旅行 Agent，等于同时维护两套地理逻辑。** 这是「境内外差异」这个大主题的一个切面——它大到值得单独一部分来讲（第五部分）。这里先记住结论：**别指望一套地理逻辑通吃全球，境内境外要分开设计、分开验证。**

## 领域知识落在哪

这一章出现的领域知识——环线要闭合、驾时有上限、坐标要落准、境内外分开——它们最后是**落在系统的不同地方**的，这恰恰预示了第四部分最后一章的主题：

- 「环线要闭合」「驾时别太长」这类**判断原则**，落进 prompt 和 critic 的核对清单；
- 「离群坐标丢弃」「非自驾不画折线」这类**确定性规则**，落进代码；
- 「估算这段驾时」这类**要模型主动调用的动作**，落进工具。

**同一套领域知识，会散落在 prompt、代码、工具三个地方，各就各位。** 怎么判断哪条知识该放哪，是「领域知识注入」那一章要专门讲的事。
