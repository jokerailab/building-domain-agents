# 领域知识注入

前面几章反复出现同一种说法：这条知识「落进 prompt」、那条规则「焊进代码」、这个动作「做成工具」、这批判断「沉进车型库」。这一章正面回答这个反复出现的问题——**领域知识到底该注入到哪里，怎么判断？**

这不是个小问题。因为**同一套领域知识，注入到不同的地方，效果天差地别**：一条本该焊进代码的确定性规则，你写进 prompt 靠模型自觉执行，它就会时灵时不灵；一条本该让模型灵活判断的软原则，你写死进代码，它就会僵硬、处理不了例外。**注入口选错了，再对的知识也发挥不出来。**

## 领域知识有四个「落点」

一个专用 Agent，领域知识可以注入四个地方，每个装的东西性质不同：

| 落点 | 装什么 | 特点 |
|---|---|---|
| **prompt** | 判断原则 | 要模型理解、灵活运用、能处理例外 |
| **工具描述** | 用法细则 | 局部、就近，只在用这个工具时相关 |
| **代码** | 确定性规则 | 硬、每次必执行、不容裁量 |
| **数据** | 可枚举的事实与判断 | 量大、可查询、可沉淀、是资产 |

下面一个个说。

## 落点一：prompt——装「判断原则」

那些**要模型理解并灵活运用**的软性原则，放进 prompt。比如：

- 「自驾环线要闭合」；
- 「带老人小孩要控制驾时」；
- 「多人结伴要撮合出大家都能接受的共识」；
- 「禁止凭空编造城市、距离、价格」。

这些的共同特点是——**它们是判断，有弹性，要模型结合具体情况拿捏。** 「控制驾时」不是一个死数字（真越野和城市周边的容忍度不同），「撮合共识」更是要模型灵活权衡。**prompt 是模型的「价值观和常识」**——你在这里注入的是「这个领域该怎么想」，而不是「第几步该干什么」。

## 落点二：工具描述——装「用法细则」

关于「某个工具什么时候用、怎么用」的规则，下沉进那个工具自己的 description（第一部分「工具即能力边界」讲透过）。比如：

- `web_search` 的「查到的结论要写进 notes——查了不写＝没查」；
- `research` 的「一批要核实的用它、别一个个 web_search」。

这些的特点是——**它们是局部的、就近的，只在模型考虑用这个工具的那一刻才相关。** 写进全局 prompt 是噪音（大部分时候用不上），跟着工具走才恰到好处。**工具描述装的是「这件具体的事怎么做对」。**

## 落点三：代码——装「确定性规则」

那些**没有弹性、必须每次都精确执行**的硬规则，焊进代码。比如：

- 离群坐标（飘到千里之外的错点）直接丢弃；
- 模型吐出的畸形结构（数组被 stringify）自动矫正；
- 单日驾时超过明显不合理的阈值，判定这天排错了；
- 跨午夜的到达时刻，标成「次日 HH:MM」。

这些的特点是——**它们是确定的，不容模型自由裁量，错了有明确且严重的后果**（崩 500、地图歪掉、时间藏错）。**这种规则绝不能交给模型「自觉」**——模型会漏、会忘、会时对时错。**确定性的事，交给确定性的代码。** 这也是第二部分「弱模型护栏」的思路：凡是不能出错的，别指望模型，用代码兜死。

## 落点四：数据——装「可枚举的事实与判断」

那些**量大、要查询、要持续沉淀**的领域知识，做成结构化数据。比如：

- 车型库（每款车的参数 + 自驾判断）；
- 目的地库（每个地方的地貌、海拔、独特度）。

这些的特点是——**它们可枚举、可扩充、是资产**（第三部分「数据护城河」讲的就是这一类）。你不会把三百款车的判断写进 prompt（塞不下、也没法查询），而是做成一个能按需检索、能持续扩充的库。**数据装的是「这个领域里那些又多又要沉淀的事实和判断」。**

## 怎么判断一条知识该放哪

拿到一条领域知识，问自己四个问题，就知道它该注入哪个口：

1. **要模型灵活运用、能处理例外吗？** → prompt；
2. **只在用某个具体工具时才相关吗？** → 那个工具的描述；
3. **必须每次精确执行、不容裁量吗？** → 代码；
4. **量大、可枚举、要持续沉淀吗？** → 数据。

**判据的核心，是这条知识「有没有弹性」和「什么时候相关」。** 有弹性、要判断的，交给模型（prompt / 工具描述）；没弹性、要死守的，交给代码；又多又要沉淀的，交给数据。

## 同一条知识，会拆到多个落点

最后一个重要的认识：**一条领域知识，常常不是只进一个口，而是按「性质」拆开，分头注入。**

拿「驾时别太长」这一条，看它怎么散落到四处：

- **原则**（排线时要考虑驾时、别让某天太赶）→ 写进 **prompt**；
- **卡位**（自驾行程提交前必须估过驾时）→ 做成 **工具** 的必经调用（estimate_leg）；
- **粗估逻辑**（怎么把两点算成一个驾时）→ 在 **工具的代码实现** 里；
- **硬守卫**（单日超过某阈值判定排错）→ 焊进 **代码**；
- **核对**（出行程前检查有没有违反）→ 交给 critic 子 Agent。

**同一条「驾时」知识，它的「原则」「动作」「计算」「守卫」「核对」被拆到了不同落点，各司其职。** 这才是领域知识注入的完整形态——不是「把这条知识放哪」，而是「把这条知识的不同侧面，分别放到最合适的地方」。

## 这就是「通用架构 × 专业能力」的接合点

到这里，可以回答开篇那个命题了——**专用 Agent = 通用 Agent 架构 × 专业业务能力，这两者到底在哪里结合？**

答案就是这一章：**通用架构，提供的正是这四个「注入口」**——prompt、工具、代码、数据。这套注入口是通用的，做任何领域的 Agent 都长这样。而**专业能力，就是往这四个口里，灌进你这个领域对的知识**——什么原则该让模型拿捏、什么规则必须死守、什么判断要沉淀成数据。

**做专用 Agent = 有一套好的注入口（通用架构给的）+ 知道什么知识该灌进哪个口（你的专业判断）。** 前者可以学、可以抄；后者是你对这个领域理解的深度，抄不走。两者结合的质量，决定了你的专用 Agent 到底有多「专业」。
