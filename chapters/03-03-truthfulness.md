# 真实性优先

前两章里，「真实性」这个词反复冒出来：核实是为了防编造、推荐不能编参数、选择不能让模型碰价格。这一章把它单拎出来讲，因为它是专用 Agent——尤其是**要帮用户做真实决策**的那种——最不能让步的一条底线。

一句话立场：**对一个旅行 Agent，编造一个不存在的航班、一个错误的价格、一条走不通的路，比「这个信息我暂时给不了」严重一百倍。** 前者会让用户照着一个假的东西去订票、去出发；后者只是少了个功能。**真实性不是一个可选功能，它是生命线。**

## 为什么真实性对这类 Agent 是生死线

通用聊天 Agent 编错一个事实，用户顶多被误导一下，代价有限。但一个旅行 Agent 不一样——**它的输出是要拿去做真金白银的决策的。**

- 编一个「东航 MU5432，￥820」的航班——用户信了，去订，发现根本没这班飞机；
- 编一个「稻城到成都 4 小时」的驾时——用户按这个排了行程，实际要开 8 小时，天黑了还在山路上；
- 编一个「XX 观景台」——用户特意绕过去，发现那地方不存在。

每一个都是**用户照着假信息行动、然后在现实里撞墙**。而且最坏的是——**模型编造的时候，语气和真实信息一模一样自信。** 用户没有办法从措辞上分辨哪句是查证过的、哪句是编的。所以一旦你的 Agent 有过一次「言之凿凿地骗了我」，用户对它**所有**输出的信任都会崩塌，包括那些真实的部分。

**信任是这类产品唯一的护城河，而信任是由真实性一砖一瓦砌起来的，一次编造就能推倒。**

## 防线一：能力上防编造（工具 + 护栏）

真实性不能靠在 prompt 里写一句「不要编造」就指望模型做到。它得靠**架构**，让模型在结构上**编不了**。这个项目里有好几道这样的防线，前面章节其实都埋过：

- **核实工具**（`research` / `web_search` / `read_source`）：把模型脑子里「觉得」的东西，逼成「查过」的。`SYSTEM_PROMPT` 第一句就是「**禁止凭空编城市 / 距离 / 价格**——一切靠工具核实」。
- **选择闭环**（`fetch_options` + `choose`）：价格、航班号从真实查询到落地，全程待在代码里，模型只透传展示文本、碰不到真实字段（第一部分讲透过）。
- **真实数据源**：机票走能返回真实航班号 / 时刻的数据源、酒店走真实 POI，都标「参考价」诚实标注——有多少确定信息，给多少。

这些合起来是一个原则：**把「不能错的东西」放到模型够不着的地方。** 模型负责判断和编排，但真实数据的经手权不交给它。这是防编造最可靠的一招——不是让模型「别编」，是让它「没机会编」。

## 防线二：诚实地面对「我给不了」

但光有防线一还不够。真实性最难的部分，不是「不编造」，而是——**当你真的拿不到某个真实信息时，怎么办？**

弱一点的产品会选择「圆过去」：拿不到真实价格，就估一个看起来像样的；拿不到真实链接，就给一个跳过去可能打不开的。这是在用「看起来完整」换掉「真实」。而这个项目的选择是相反的：**宁可少给，也不假给。**

这条原则在「跳转链接」这件事上体现得最彻底，也最能说明问题。一个旅行 Agent 排完行程，总想给用户「一键去订」的链接。但真实情况是——**很多跳转链接是靠不住的**：有的跳过去要登录才能看、有的把中文地名当搜索词结果跳回首页、有的境内根本打不开。

面对这个，做法不是「有链接就放上去」，而是**把每一种跳转都实际验证一遍**，然后按结果分两类处理：

- **真的能用的**（免登录、能出真实结果的深链）——做成精准的「去订」入口；
- **不能用的**（跳登录墙、跳回首页、境内打不开的）——**一律删掉**，退回成「展示信息 + 诚实提示」，比如「去 XX App 搜『地名』下单」。

**「跳过去打不开」比「没有这个按钮」更糟——因为它让用户对整个产品的可信度打折。** 一个假的「一键预订」，暴露的是「这个产品的功能是虚的」；而一句诚实的「这个我给不了精准链接，你去 App 搜一下」，反而让用户觉得「它不糊弄我」。

这是一个反直觉、但我认为极其重要的产品判断：**在真实性和功能完整度冲突时，选真实性。** 少一个功能，产品只是「小一点」；多一个假功能，产品是「假的」。用户能原谅前者，不能原谅后者。

## 「看不到＝没竞争力」和「不能编」怎么共存

这里有个张力值得点破。这个项目的定位是**信息聚合器**——关键决策信息（有哪些酒店、什么航班、什么车）要让用户在对话和报告里**直接看到、能比、能选**，看不到就没竞争力。但同时又立着「绝不编造」的铁律。这两条看起来矛盾：既要信息丰富，又不许编。

解法是一个很关键的洞察：**把「依赖实时数据的信息」和「不依赖的信息」分开。**

- 「有哪些酒店、评分怎么样、什么车型、有哪些航班」——这些**不依赖具体日期**，用真实数据源随时能聚合，尽管给足；
- 「今天这个房间多少钱、这班机票多少钱」——这些**只有给了日期才查得到真实值**。没有日期时，就**引导用户补日期**（「有日期我才能查到真实房价」），而不是编一个假价格填上去。

**信息可以尽量丰富，但每一条都必须是真的；拿不到的那条，用引导去补，而不是用编造去凑。** 「丰富」和「真实」不矛盾——矛盾的是「丰富」和「偷懒」。真实性不是让你少给信息，是逼你用更诚实的方式（真实数据源 + 引导补全）去把信息做丰富。

## 对专用 Agent 意味着什么

如果你的 Agent 要帮用户做真实世界的决策（订票、投资、诊断、报价……），把这一章的三条刻进去：

1. **真实性是生命线，不是功能点**——一次自信的编造，摧毁的是对全部输出的信任；
2. **防编造要靠架构，不靠 prompt 恳求**——把不能错的数据放到模型够不着的地方；
3. **拿不到真实信息时，诚实地说「给不了」**——宁可功能少一点、用引导补全，也绝不用假信息凑完整。

**用户要的从来不是一个「看起来什么都能干」的 Agent，是一个「说的话能信」的 Agent。** 后者才走得远。
