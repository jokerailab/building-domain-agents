# 并发闸门

上一章让单个用户的一趟行程能稳稳跑完。但平台是**多用户**的——十个人同时点「开始排行程」，会发生什么？这一章讲的，是一个免费产品在模型这一层最容易翻车、又最容易被忽略的地方。

## 最危险的失败：所有人一起悄悄变差

先说清楚这个坑长什么样，因为它不报错，所以特别隐蔽。

平台的免费用户，共享一个模型池（上一章说的 `user-pool`）。这个池是**免费额度堆出来的**，容量有限。而排一趟行程，是**跨好几分钟、许多轮的大请求**——一次设计就能吃掉不少额度。所以这个池同时稳得住的并发设计数量，其实很少——**只有个位数、甚至一两个**。

那第三个、第四个用户同时进来呢？请求撞上限流，网关的 fallback 链就开始一级级往下降——降到链尾那个又弱又糙的兜底模型。而上一章讲过：**弱模型驱动不了 function-calling**，排行程的工具循环在它手里会卡死、出不了报告。

最要命的是最后半句：**用户对这一切毫无感知。** 他不知道自己被降级到了弱模型，只觉得「这个 AI 今天怎么这么慢、这么笨、半天排不出来」。他不会怪「并发太高」，他会怪你的产品。

这里有个关键的认知纠偏，我一开始也没想明白：**真正的问题，不是「免费额度用完了」。** 额度用完有 fallback 兜着，流程不会硬崩。真正的问题是——**高并发下的静默降级 + 用户零反馈**。系统在悄悄地把体验劣化，而用户被蒙在鼓里。**一个会无声变差的系统，比一个会明确报错的系统更伤用户信任**，因为它让用户在不知情的情况下，把「系统过载」误读成「产品无能」。

## 闸门：与其一起变差，不如排队等好的

想通了问题是「静默降级」，解法就清楚了：**与其让所有人一起挤进来、一起悄悄降级到弱模型，不如让一部分人排队等着，但每一个真正跑起来的，都用的是好模型。**

这就是并发闸门。实现上，是在 SSE 流的外面套一层 gate：

```python
# 伪代码：闸门套在 agent 流的外面
_DESIGN_MAX = 2                       # 同时最多 2 个设计在跑（按额度调）
_sem = threading.Semaphore(_DESIGN_MAX)

def gate_events(make_events, gated):
    if not gated:                     # BYOK / admin 不受闸
        yield from make_events(); return
    if not _sem.acquire(blocking=False):
        # 满了 → 排队，每 ~3s 给前端一个反馈事件（同时当心跳）
        while not _sem.acquire(timeout=3):
            yield {"type": "queue", "ahead": queue_depth()}
    try:
        yield {"type": "queue_ready"}
        yield from make_events()      # 拿到令牌，此刻才真正启动 agent
    finally:
        _sem.release()
```

有两个设计细节，是这个闸门好用的关键：

**一、排队期间，绝对不碰 LLM。** 注意 `make_events` 是一个**延迟工厂**（lambda），排队的时候它根本没被调用——也就是说，排队中的用户**一次模型请求都不会发出**。这很重要：如果排队时还在打 LLM，那闸门就白设了，池子照样被打爆。闸门的意义就在于「拿到令牌之前，一个 token 都不花」。

**二、排队要给反馈。** 每隔约 3 秒，往 SSE 流里推一个 `queue` 事件（顺便当心跳，防连接被中间层掐断），前端显示一张排队卡：「前面还有 N 个在排」。拿到令牌了，推一个 `queue_ready`，前端清掉排队卡、无缝接上正式的排行程流。

**这一层的全部意义，就是把「静默降级」换成「可见的等待」。** 用户宁可看到「你在排队，前面 2 个」这样诚实的等待，也不要对着一个不知为何变笨的 AI 干瞪眼。**明确的等待是可以接受的，无声的劣化不可以**——这是并发设计里一条关于用户心理的底层判断。

## 谁该被闸：把限流和商业模式缝在一起

闸门不是拦所有人。看那个 `gated` 判断——**只有平台免费用户走闸门**：

```python
gated = (user_provider is None) and (not is_admin)
```

用户自己配了 key 的（BYOK），用的是他自己的模型额度，跟平台的免费池毫无关系——**不占池、不排队、不受限**。管理员同理。

这个判断把两件事缝到了一起：**技术上的限流，和商业上的分层。** 免费用户共享有限的池子，所以要排队；付费自带 key 的用户走自己的资源，所以畅通无阻。限流不再是一个纯技术措施，它同时定义了「免费和付费的体验差在哪」。

而这就引出了闸门最妙的一个副作用——

## 一个限流措施，同时是增长入口

排队卡上，除了「前面还有 N 个」，还有一行字：**「接入你自己的 Key，免排队 →」**，链接到设置页。

这一下，闸门从一个「防过载的技术措施」，变成了一个「把重度用户转化为自带成本用户的增长入口」。逻辑非常顺：**最可能碰到排队的，恰恰是用得最频繁、最重度的用户——也正是最有动力、最值得引导去 BYOK 的那批人。** 你不需要弹窗推销，用户在真实感到「不够快」的那一刻，你恰好给出「有个办法能立刻变快，而且成本归你」的出口。

**一个好的限制，本身就能讲出一个产品叙事。** 免费池的容量限制是客观存在的成本约束，但通过「排队 + BYOK 引导」这个设计，它从一个纯粹的负面约束，变成了「免费够用、重度自带」这个清晰商业模式的自然承载。约束没变，但它被用出了正向的价值。

## 排队本身也要有上限

最后一个不那么显然、但不做会埋雷的点：**排队队列本身也得封顶。**

为什么？因为每一个排队中的 SSE 连接，都实实在在地占着服务器的一个线程（长连接）。在一台不大的、单 worker 的服务器上，线程是有限的——如果排队没有上限，一次流量高峰能让**排队的人**就把线程池占满，进而拖垮整个站点（包括那些只想看看广场的用户）。限流是为了保护系统，结果限流的排队区反而成了新的过载源，就本末倒置了。

所以队列有个硬上限（比如 20）。排到第 21 个，不再让他傻等，而是直接返回一个「现在太忙了」的友好错误——同样带上那句 BYOK 引导。这里还有个并发编程的小坑值得一提：这个「拒绝」的反馈，必须在**释放锁之后**再往外推（yield），否则容易和信号量的获取/释放绞在一起死锁。

这一整套——**限并发、给排队反馈、封排队上限、引导 BYOK**——合起来才是一个完整的闸门。它保护的不只是模型池，是整台服务器在流量高峰下的「不崩」。

（补一句边界：闸门只拦**首轮**的排行程请求，不拦续聊 / 恢复的请求。因为一个已经排到一半、正在跟用户来回的会话，中途被塞进队列去排队，体验是灾难性的——交互的连续性，优先于严格的限流。）

> **一个策略层面的后续（重要）**：这一章把「排队 + BYOK 引导」讲成一个把免费用户升级成 BYOK 的「增长入口」——那是免费模型还对普通用户开放的阶段。后来这个产品做了个更进一步的判断：**免费模型的体验对垂类 Agent 就是不够好，干脆让排线一律走 BYOK，免费池收敛成只给白名单的内测通道**。于是本章的「排队闸门」主要服务那个受控的内测池，而「BYOK 引导」也从一个「增长入口」升级成了核心动作的「主门槛」。为什么这么转，是这一部分最后一章的主题。

> 闸门保证了「真正跑起来的，都是好模型」。但百密一疏——万一还是有请求降级到了弱模型呢？弱模型不会优雅地承认「我不行」，它会一本正经地把事情搞砸：断流、死循环、编造。下一章讲**弱模型护栏**：怎么检测降级、怎么强制收尾、怎么让最坏情况下的失败，至少是「体面且可恢复」的。
