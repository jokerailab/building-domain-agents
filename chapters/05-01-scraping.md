# 抓取架构

上一部分反复强调「真实数据源」。但「拿到真实数据」本身，是一个不小的工程活——尤其当你要**大规模、持续地**抓，而手上又只有很有限的资源。这一章讲抓取的架构，以及它背后一个反直觉的第一约束。

## 第一约束：绝不被封

先说清楚这个约束，因为它决定了整个抓取架构的形态。

抓取要通过一个**出口 IP** 去访问目标网站。而现实是——**这个 IP 是共享的、有限的，一旦因为抓取行为被目标站封了，整条数据管道就断了。** 对一个小团队，你没有一大片 IP 池可以轮换、也没有预算买昂贵的代理服务。你就那么一个（或很少几个）出口。

这就带来一个和大厂完全不同的优先级：**抓取架构的第一约束，不是「抓得快」，是「绝不被封」。** 快慢是效率问题，被封是生死问题——被封一次，你今晚的数据就没了，可能好几天缓不过来。所以整个抓取系统，是围绕「稳」而不是「快」来设计的。

**认清你真正的约束是什么，比照搬大厂的做法重要得多。** 大厂的抓取架构追求吞吐、并发、IP 轮换；而你的现实是「一个 IP 不能死」——这两种约束下，架构长得完全不一样。

## 全局抓取锁：所有浏览器抓取，永远串行

围绕「绝不被封」，最核心的一个设计是**一把全局锁**。

这个系统里，所有需要用浏览器去抓的任务——夜间定时抓取、数据回填、新目的地发现、手动补数据——**共享同一把锁**。这把锁保证：**同一时刻，最多只有一个浏览器在碰那个出口 IP。** 永远串行，绝不并发。

```python
# 所有浏览器抓取任务，进来先抢这把锁
with scrape_lock("nightly-scrape", wait=True):
    ...   # 抓取；同一时刻别的浏览器任务在锁外排队
```

为什么是这样？因为**并发抓取是被封的头号原因**——同一个 IP 同时开五个浏览器猛抓，目标站一眼就识别成机器行为。串行慢，但安全：一个接一个、有节奏地抓，看起来更像正常访问。

几个实现上的讲究：

- **它是一把文件锁**，跨进程有效（不同的定时任务是不同进程）；
- **能识别「僵尸锁」**——如果持锁的进程崩了、或者持锁超过一个很长的时限还没释放，后来者可以「偷」走这把锁，防止一次崩溃把整条管道永久卡死；
- **纯 API 的任务不受这把锁管**——那些不走浏览器、只调正规 API 的数据获取（比如机票的那个免费 API），走的是另一条不易被封的路，没必要跟着串行排队。

**这把锁的本质，是用「串行的慢」换「永不被封的稳」。** 对一个靠单 IP 的小团队，这个取舍是对的——抓得慢一点，用户几乎无感（数据是夜里更新的）；被封一次，代价是整条管道断掉。**在你最脆弱的资源上，宁可牺牲效率，也要买断风险。**

## 分层抓取：把力气花在刀刃上

既然抓取是串行的、总量有限的，那就不能对所有数据一视同仁地下重手——**得分层，把有限的抓取预算，花在最重要的数据上。**

这个项目给目的地分了层：**精选的核心目的地**，用浏览器全量深抓（数据最全、最准）；**长尾的目的地**，用轻量的 API 补一补就行（够用即可）。核心的那批是用户最常去、最需要准确数据的，值得下重手；长尾的抓个大概就行，不值得占用宝贵的串行抓取时间。

**抓取资源是稀缺的（串行 + 防封），所以要像分配预算一样分配它**——先喂饱最重要的数据，长尾的用更便宜的方式覆盖。一视同仁地深抓一切，是对稀缺抓取能力的浪费。

## 增量校准：抓取是「补」，不是「重建」

还有一个容易被忽略、但很重要的原则：**抓取是给已有数据做增量校准，不是每次推倒重来。**

前面讲护城河时说过，车型库的判断层（评分、适合 / 慎选、结论）是人工注入的资产。那抓取脚本去补数据时，会不会把这些辛苦攒的判断覆盖掉？

不会——因为抓取默认是「**只补空**」：只填那些空着的**客观事实**字段（离地间隙、涉水深度这些），**人工注入的判断字段，一个都不碰。** 只有显式加一个「覆盖」参数，才会更新已有的事实数值。

这个「只补空」的默认值，保护的是你最值钱的东西：**抓取补的是可以自动化的事实，绝不能顺手抹掉不能自动化的判断。** 抓取是给资产做增量维护的工具，不是重置资产的推土机。把这个默认设对，你才敢放心地反复跑抓取脚本，而不用担心哪次跑完把护城河冲掉了。

## 选反爬最轻的源

同一份数据，往往有多个来源，而它们的**反爬难度天差地别**：有的源要登录、还对数字做混淆，抓起来又难又脆；有的源不用登录、数据就是干净的文本，抓起来又稳又省。

**在数据一样的前提下，选反爬最轻的那个源。** 这不是偷懒，是理性——你的目标是拿到真实数据，不是跟反爬机制较劲。为一个更难抓的源投入大量精力去对抗它的反爬，通常不划算，尤其是当有一个更好抓的源能给你同样的数据。

还有一个心态很重要：**别在反爬上无止境地烧时间。** 这个项目的数据库是**独立可用**的——就算某次抓取失败，已有的数据照样服务用户，抓取只是增量补充。想清楚这一点，你就不会为了「这次一定要抓下来」而陷进跟反爬的军备竞赛里。**抓取是锦上添花，不是命根子——你的命根子是那个已经能独立服务的数据库。**

## 夜里串一条链跑

最后，这些重活——抓取、回填、发现新目的地、清理——都放在**夜间定时**执行，而且**串成一条链、顺序跑**。夜里用户少，慢一点没人在意；串行执行，天然就不会有并发冲突，也自然服从了那把全局锁的约束。**把重活挪到低峰期、排成一条串行的链，是小系统用最少的复杂度换最大的稳。**
