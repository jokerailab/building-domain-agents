# 记忆系统

上一章讲上下文，是**短期记忆**——一次会话里窗口装着的东西，会话一结束就没了。但一个真正好用的专用 Agent，不该每次都从零开始：老用户回来排第三趟行程，它不该还在问「你喜欢住民宿还是酒店」——这个你半年前就告诉过它。

**记忆系统就是 Agent 的长期记忆：跨会话地记住「这个用户是谁」。** 在 Claude Code 里，这个东西叫 `CLAUDE.md`——你的项目约定、代码风格、常用命令，写一次，之后每次它都记得。野行 Y 里对应的东西，是**用户画像**。

## 记忆和上下文的分工

先厘清两个概念，它们经常被混为一谈：

| | 上下文（短期） | 记忆（长期） |
|---|---|---|
| 存在哪 | 模型的窗口里 | 数据库 / 文件 |
| 活多久 | 一次会话 | 跨会话、永久 |
| 谁写 | 对话和工具自动填 | 显式沉淀下来 |
| 怎么用 | 模型直接读 | **提炼后注入**上下文才能用 |

关键在最后一行：**记忆本身模型是看不见的。** 数据库里躺着一个用户的偏好，模型不会「感应」到——必须在合适的时机，把它提炼成一小段，注入到这次会话的上下文窗口里，模型才读得到。所以记忆系统和上下文工程是接力关系：记忆负责**长期存**，上下文负责**当次用**。上一章埋的那个「接缝」，就是这里。

## 野行 Y 记两种东西

这个项目的用户画像（`build_user_profile`，在 `trips_social`）拼两类信息：

**一、显式偏好**——用户在账户里设过的：住宿档次、出行节奏、预算档、体力、忌讳、兴趣。这些是他主动告诉系统的，最可靠。

**二、行为记忆**——他做过 / 参与过的行程。这个不用问，从数据库里捞：

```python
# 他自己排的行程，UNION 上他作为队员参与的别人的行程，取最近 6 条标题
rows = c.execute(text(
    "SELECT title FROM ("
    "  SELECT data->>'title' AS title, created_at FROM itineraries WHERE owner_id=:u "
    "  UNION "
    "  SELECT i.data->>'title', i.created_at FROM itineraries i "
    "  JOIN trip_participants p ON p.trip_id=i.id WHERE p.user_id=:u AND p.role<>'owner'"
    ") t ORDER BY created_at DESC LIMIT 6"), {"u": uid})
```

于是一个老用户开新对话，Agent 一上来的 system 里就带着：「固定偏好：住民宿 · 深度慢玩 · 预算适中；做过 / 参与过的行程：川西小环线；洱海环湖；……」。它一张嘴就「记得」这个人——「你之前走过川西，这次给你换个方向」。这是 Claude Code 读 `CLAUDE.md` 那种「它好像懂我」的体感，搬到了旅行领域。

**做专用 Agent 时，你要回答的问题是：这个领域里，什么值得跨会话记住？** 写代码要记的是项目约定和代码风格；排行程要记的是「这个人怎么玩、去过哪、忌讳什么」。答案完全是领域相关的，没有通用答案——这正是「专用」的一部分。

## 记忆的第一个坑：记岔了

记忆最危险的地方，不是没记住，是**记错了还理直气壮**。模型对注入进来的东西是全盘相信的——你往它记忆里塞一句错的，它会拿这句错的去误导用户整趟行程，而且语气笃定。

所以这个项目里有一条铁律，从架构第一天就立着：**记忆里的每一条，都必须是真实、可追溯的，绝不能是模型自己编出来的。**

看上面那段代码——「做过的行程」是从 `itineraries` 表里真实查出来的标题，不是让模型「回忆一下这个用户可能去过哪」。偏好是用户在账户里真实设过的，不是模型猜的。这跟整个 Agent 的第一性原则是一致的：`SYSTEM_PROMPT` 开头就写着「**禁止凭空编城市/距离/价格**」——这条对工具成立，对记忆同样成立。

**一个会「编记忆」的 Agent，比一个没记忆的 Agent 更糟。** 没记忆最多是每次重新问一遍，烦一点；编记忆是拿着假的「我记得你说过」去误导决策，用户还不容易察觉。所以记忆系统的可信度，优先于它的丰富度。

代码里还有一个小而关键的细节：新用户（没有任何数据）时，`build_user_profile` 返回**空字符串，什么都不注入**。宁可不注入，也不硬凑一段「这个用户可能……」的猜测。**没有记忆，好过错的记忆。**

## 记忆的第二个坑：生搬硬套

第二个坑更微妙。就算记忆是真的，把它当成「命令」来执行，也会出问题。

一个用户去过川西，不代表他这次也想去高原；他偏好住民宿，不代表这次带着老人也非民宿不可。记忆是**背景**，不是**指令**。如果 Agent 把「他上次住民宿」当成硬约束，这次给带老人的行程也硬塞民宿，那就是被记忆绑架了。

所以注入用户画像时，我在提示语里明确给它松了绑：

> ## 关于这位用户（背景，帮你更懂他——**别生搬，按当前这次的需求灵活参考**）

这一句「别生搬」很重要。它告诉模型：这些是帮你更懂他的背景色，不是这次任务的硬指标。**记忆的正确用法是「参考」，不是「服从」。** 当前这次对话里用户新说的话，永远优先于历史记忆——就像 Claude Code 里 `CLAUDE.md` 是默认约定，但你这次明确说「这个文件别管那些规范」，它就该听你当下的。

这背后是一个更普遍的判断：**记忆的权重要低于当次意图。** 长期记忆解决的是「默认值」问题（不用每次从零问起），但它不该盖过用户此刻的明确表达。把这个优先级搞反，Agent 会显得又固执又不听话——明明记性好，体验却更差。

## 小结：记忆系统的三条准则

做专用 Agent 的记忆系统，我把它收敛成三条：

1. **只记这个领域里真正跨会话有价值的东西**（旅行：偏好 + 去过哪 + 忌讳），别什么都往里塞；
2. **每一条都必须真实可追溯**，从数据库来、从用户显式设置来，绝不让模型编——宁可空着；
3. **注入时明确它是「背景参考」而非「硬指令」**，当次意图永远优先。

记住这三条，记忆会让 Agent「越用越懂你」；违反任何一条，它会「越记越坑」。
